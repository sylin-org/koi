# Reader: adapters-ux

## Summary

Koi's transport layer is four adapters in the binary crate (HTTP, mTLS, Named Pipe/UDS "IPC", piped-stdin "CLI") plus two embedded web surfaces (dashboard, mDNS browser) whose actual implementation now lives in koi-common, not the binary crate as the project docs claim. The web UX is built from two self-contained single-file HTML/CSS/JS pages embedded via include_str! (crates/koi-common/assets/dashboard.html, 698 lines/29.9KB; mdns-browser.html, 720 lines/30.2KB) — vanilla JS, no build step, shared design tokens, dark mode. Server-side, koi_common::dashboard (159 lines) is a clean domain-agnostic adapter (SnapshotFn closure + broadcast SSE + heartbeat) and koi_common::browser (563 lines) maintains a capped discovery cache fed by a meta-browse worker that spawns per-type browse pumps. The dashboard shows capability cards, per-domain detail panels, an OpenAPI-driven API reference, and a live SSE activity log; it is strictly read-only (no mutation controls). The mDNS browser — filterable, sortable, with TXT-record detail panes and inferred "launch" links for HTTP-ish services — is the strongest UX artifact and genuinely useful to a homelab user.

The HTTP adapter (crates/koi/src/adapters/http.rs, 834 lines) is the most mature piece: per-domain route mounting with a tested 503 capability_disabled fallback router, OpenAPI composed by nest()-ing seven domain ApiDocs plus system schemas, Scalar UI at /docs, a Daemon Access Token middleware using constant-time comparison (mutations only; GET/HEAD/OPTIONS exempt), careful middleware-ordering comments, and loopback-only binding with a separate 0.0.0.0 mTLS plane (adapters/mtls.rs) for certmesh inter-node traffic. Its main wart is the unified_status_handler: ~110 lines of seven hand-written if/else blocks that are duplicated, with drift, in at least three other places (adapters/dashboard.rs build_snapshot_value, commands/status.rs offline_capabilities, koi-embedded lib.rs) because the Capability trait is sync while several statuses are async — three different call conventions coexist for the same concept.

The pipe and CLI adapters are confirmed mDNS-only: the shared dispatch (adapters/dispatch.rs) handles exactly the six mDNS verbs, the IPC adapter is skipped when mDNS is disabled (main.rs:742-755), and koi-client speaks only HTTP — there is no first-party consumer of the pipe. The one documented use case is mounting the Unix socket into containers for socat NDJSON with session-scoped leases (CONTAINERS.md:665), which is a genuinely nice lifecycle primitive (connection drop → 30s grace → unregister) that HTTP heartbeats only approximate. At ~430 lines including tests the IPC pillar is cheap, but it covers 1 of 8 domains, adds a third documented protocol surface, and sits in an awkward half-state: too narrow to be a real pillar, too entrenched (docs, tests, container guide) to be free. The biggest structural issue in this lens is that moving the dashboard/browser into koi-common turned the "shared kernel" into a presentation host — it now depends on axum, async-stream, chrono, tokio-stream, and hostname, which every domain crate transitively carries — while still failing to eliminate duplication: koi-embedded re-implements the BrowseSource adapter and the dashboard event forwarder nearly verbatim, and the two forwarder copies have already diverged (embedded forwards runtime events; the daemon's does not). Finally, the browser page has concrete XSS vectors from LAN-attacker-controlled mDNS data: esc() does not escape quotes yet values are interpolated into HTML attributes, and inferEndpoint accepts javascript: URLs from TXT records.

## Strengths

- Disabled-capability 503 fallback router is clean, uniform across all 7 domains, and unit-tested (crates/koi/src/adapters/http.rs:692-757) — runtime capability tunables are honored consistently at the HTTP layer
- OpenAPI composition via utoipa nest() of domain-owned ApiDocs with per-tag external doc links, Scalar UI at /docs, pre-serialized spec, and tests asserting both system and domain paths exist (http.rs:324-449, 759-804) — strong API DX for a homelab tool
- DAT auth middleware uses constant-time comparison via the subtle crate, exempts only safe methods (GET/HEAD/OPTIONS with an explicit comment about axum HEAD auto-matching), and is layered before CORS with the ordering rationale documented (http.rs:211-234, 458-495)
- Security-conscious bind split: plain HTTP on 127.0.0.1 only, LAN traffic via a separate mTLS adapter with client-cert CN extraction through a manual tokio-rustls accept loop, including unit tests for CN parsing (adapters/mtls.rs:30-110, 163-189)
- Zero-build-step web UI: two self-contained HTML files with shared design tokens, dark-mode support, SSE + polling resilience (disconnected banner, auto-reconnect), embedded via include_str! — no node toolchain, no asset pipeline, fits the single-binary philosophy
- The mDNS browser is a genuinely good UX surface: type histogram with click-to-filter, sortable/filterable instance table, expandable TXT-record detail panes, and inferred launch buttons for HTTP services from type/port/TXT hints (mdns-browser.html:392-449) — comparable to standalone discovery apps
- koi_common::dashboard is a well-factored inversion: zero domain logic, snapshot injected as a boxed async closure, events as a plain broadcast channel (koi-common/src/dashboard.rs:1-68) — the abstraction itself is sound even if its crate placement is not
- Pipe/CLI adapters share NDJSON dispatch exactly as documented (adapters/dispatch.rs), each adapter keeping only transport setup and its session grace constant; dispatch and write_response have real test coverage including serialization-failure fallback (dispatch.rs:116-128)
- Session-scoped lease semantics on the IPC adapter (connection drop → 30s grace → unregister, pipe.rs:13,80) are the cleanest container lifecycle story in the product — the OS reports the disconnect, no heartbeat loop needed
- BrowserCache has sensible resource bounds: 2000-instance cap with oldest-eviction, 120s purge of removed entries, lag-tolerant SSE streams (koi-common/src/browser.rs:115-118, 220-252, 499-503)

## Concerns

- XSS from LAN-controlled mDNS data in the browser page: esc() (textContent→innerHTML) does not escape double quotes, but values are interpolated into HTML attributes — data-key/data-detail at crates/koi-common/assets/mdns-browser.html:563,574, data-type at :477,500, title at :559. A malicious LAN device announcing a service named '" onmouseover="...' achieves attribute injection → inline-handler XSS on the dashboard origin
- inferEndpoint validates TXT url= values only with new URL(), which accepts javascript: scheme (mdns-browser.html:419-424) and renders it as a clickable launch button href (:558) — a second LAN-sourced script-execution vector; scheme allowlist (http/https) is missing
- koi-common, documented as the shared kernel ('types, errors, pipeline, id, paths, ceremony'), now hosts 60KB of HTML assets plus a 563-line browse cache/worker and SSE streaming, dragging axum, async-stream, chrono, tokio-stream, hostname into every domain crate transitively (koi-common/Cargo.toml:21-28) — a significant layering inversion
- Duplication the koi-common move was supposed to solve still exists and has already diverged: koi-embedded/src/mdns_browse_adapter.rs (100 lines) is a near-verbatim copy of koi/src/adapters/mdns_browser.rs (113 lines), and koi-embedded's inline dashboard event forwarder (lib.rs:446-490+) duplicates adapters/dashboard.rs:350-450 — but the embedded copy forwards runtime events while the daemon copy does not (no runtime/udp arms in spawn_event_forwarder), so the daemon dashboard activity log silently misses runtime lifecycle events
- Capability status enumeration is copy-pasted four times (http.rs:509-623 unified_status_handler ~110 lines; adapters/dashboard.rs:102-344 build_snapshot_value ~240 lines; commands/status.rs offline_capabilities; koi-embedded build_embedded_snapshot) with three different call conventions — core.status(), Capability::status(runtime.as_ref()), capability_status().await — because the sync Capability trait (koi-common/src/capability.rs:13-16) cannot express async statuses and is therefore bypassed
- Docs/code drift: .agentic/CONTEXT.md and .agentic/reference/utilities.md state dashboard/browser are 'presentation adapters in the binary crate' and locate DashboardState in adapters/dashboard.rs — both moved to koi_common (commit 941bfdd); docs/reference/architecture.md:141 claims adapters are '~150 lines each' while http.rs is 834 lines
- unified_status_handler declares ToSchema body UnifiedStatusResponse but actually returns hand-built Json<serde_json::Value> (http.rs:506-509, 616-622) — the OpenAPI schema and the real response can drift with no compile-time check
- All GET endpoints are unauthenticated by design (DAT covers mutations only) — on loopback this exposes certmesh status/audit log/roster reads to any local process and, via CORS allowing any http://localhost origin (http.rs:230-233), to any web app a user runs locally; combined with the browser-page XSS this is an exfiltration path on a tool that hosts a private CA
- Piped-stdin mode always spins up a local MdnsCore even when a daemon is running (main.rs:429-440), so two mDNS responders can coexist on one host with no coordination — piped mode never proxies to the daemon
- Stale #[allow(dead_code)] on the mtls module with comment 'Wired from main.rs when certmesh TLS material is available' (adapters/mod.rs:6-7) — mtls::start is in fact wired at main.rs:719 and platform/windows.rs:676, so the allowance masks genuine dead-code detection
- Dashboard panel data uses 3s polling while SSE feeds only the activity log (dashboard.html:340, 547-563) — two liveness mechanisms where one (SSE-triggered snapshot refresh or snapshot deltas) would do; the activity log is also volatile (50 entries, lost on refresh)
- The browser worker browses every discovered service type on the network permanently whenever mDNS is enabled in daemon mode (koi-common/src/browser.rs:362-448 wired unconditionally at main.rs:680-692) — chatty multicast behavior that the user never opted into and cannot disable separately from mDNS itself

## Maturity signals

- HTTP adapter is high-beta: graceful shutdown via CancellationToken, tested 503 fallbacks, OpenAPI path assertions, constant-time token comparison, explicit middleware-ordering comments (http.rs:211-220, 708-834)
- mTLS adapter is functional with unit tests but carries a stale module-level dead_code allowance (adapters/mod.rs:6) — a transitional, recently-wired state
- Dashboard/browser are polished-alpha: visually finished and useful, but read-only, with XSS vectors from untrusted LAN input, volatile activity logs, and dual polling+SSE liveness — presentation maturity outruns security hardening
- IPC/pipe adapter is feasibility-grade: works on both platforms, dispatch logic tested, but serves only 1 of 8 domains, has no first-party client (koi-client is HTTP-only ureq), and its sole documented consumer is a socat one-liner in CONTAINERS.md:665
- Documentation drift in three places (.agentic/CONTEXT.md adapter locations, utilities.md DashboardState location, architecture.md '~150 lines each' claim vs 834-line http.rs) indicates refactoring velocity exceeding doc upkeep — typical of a feasibility-phase project moving fast
- Three near-identical copies of event-forwarding select! loops (daemon, embedded) and four copies of capability enumeration that have already behaviorally diverged (runtime events) — duplication-driven divergence is the classic pre-beta architecture signal
- The git history shows the dashboard was moved to koi-common specifically to enable koi-embedded (commit 941bfdd), then a follow-up removed 8 Cargo deps (aa979d4) — active architectural experimentation, not settled design

## Shed or simplify

- Move dashboard/browser out of koi-common into a dedicated presentation crate (e.g. koi-dashboard) depended on only by koi and koi-embedded. koi-common returns to a types-only kernel and sheds axum/async-stream/chrono/tokio-stream/hostname from every domain crate's transitive closure (koi-common/Cargo.toml:21-28). This also restores truth to the documented architecture.
- Deduplicate the BrowseSource adapter and dashboard event forwarder between koi and koi-embedded — koi-embedded already depends on all domain crates, so a single shared implementation (in the proposed koi-dashboard crate) eliminates ~200 duplicated lines and fixes the runtime-events divergence (koi/src/adapters/mdns_browser.rs vs koi-embedded/src/mdns_browse_adapter.rs; adapters/dashboard.rs:350-450 vs koi-embedded/src/lib.rs:446-490+)
- Replace the four hand-written capability enumerations with one registry: make Capability an async trait (or add an async_status method), have DaemonCores expose iter() -> Vec<(name, Option<&dyn Capability>)>, and derive unified_status_handler, build_snapshot_value, offline_capabilities, and the embedded snapshot from that single source. Eliminates ~400 lines of if/else boilerplate and the three divergent call conventions.
- Decide the IPC pillar's fate: either (a) retire pipe.rs + cli.rs + dispatch.rs (~430 lines, 3 docs surfaces) and serve containers via HTTP with a session-lease equivalent (SSE-connected registration or lease-on-connection), or (b) commit to NDJSON as a real pillar and extend dispatch beyond mDNS. The current 1-of-8-domains half-state is the worst position — it costs documentation, testing, and a Windows Named Pipe code path while delivering value HTTP mostly duplicates. Given zero first-party consumers, (a) is the lean choice; the session-lease semantics are the only thing worth preserving.
- Fix esc() to escape quotes (or build DOM nodes and assign via textContent/dataset instead of innerHTML string concatenation) and allowlist http/https schemes in inferEndpoint — small, surgical fixes to mdns-browser.html that close the LAN-XSS vectors
- Drop the dashboard's 3s polling in favor of SSE-triggered snapshot refresh (or just keep polling and drop the SSE activity log buffering client-side) — one liveness mechanism instead of two (dashboard.html:340, 547-563, 566-620)
- Return the typed UnifiedStatusResponse from unified_status_handler instead of serde_json::json!() so the declared OpenAPI schema is compiler-enforced (http.rs:506-622)
- Remove the stale #[allow(dead_code)] on adapters::mtls (adapters/mod.rs:6) and update .agentic/CONTEXT.md, utilities.md, and architecture.md to reflect the koi-common dashboard/browser location and real adapter sizes
- Make the always-on network-wide browser worker opt-in or lazy (start meta-browse on first dashboard/browser page hit) — a daemon that permanently browses every service type on the LAN is surprising default behavior (main.rs:680-692)

- Embedded web UI: crates/koi-common/assets/dashboard.html = 698 lines / 29,890 bytes; mdns-browser.html = 720 lines / 30,154 bytes; both compiled in via include_str! (koi-common/src/dashboard.rs:26, browser.rs:34)
- Dashboard/browser server logic lives in koi-common, not the binary crate: koi-common/src/dashboard.rs (159 lines) and browser.rs (563 lines); moved there by commit 941bfdd 'move dashboard/browser to koi-common and add HTTP mDNS announcement'
- koi-common ('shared kernel') depends on axum, tokio, tokio-stream, tokio-util, async-stream, chrono, hostname, utoipa (koi-common/Cargo.toml:15-28)

- HTTP adapter binds 127.0.0.1:5641 only; mTLS adapter binds 0.0.0.0:5642 (DEFAULT_MTLS_PORT, mtls.rs:24) and mounts only certmesh inter-node routes with per-connection ClientCn Extension injection (mtls.rs:41, 94)
- Capability enumeration duplicated 4x: http.rs:509-623 (~110 lines), adapters/dashboard.rs:102-344 (~240 lines), commands/status.rs:80+ (offline variant), koi-embedded/src/lib.rs (build_embedded_snapshot + forwarder); daemon forwarder covers mdns/health/dns/certmesh/proxy but omits runtime and udp (dashboard.rs:350-357), while the embedded copy includes runtime (koi-embedded/src/lib.rs:452)
- BrowserCache bounds: MAX_INSTANCES=2000, PURGE_AFTER_SECS=120, purge sweep every 30s; SSE heartbeats every 15s on both dashboard and browser streams (browser.rs:115-118, 376, 458; dashboard.rs:89)
- Dashboard UI polls /v1/dashboard/snapshot every 3,000ms and keeps max 50 activity-log entries; browser UI polls every 5,000ms with max 30 log entries (dashboard.html:340-341, mdns-browser.html:354-355)
- esc() helper escapes only & < > (textContent→innerHTML), yet is used inside double-quoted HTML attributes at mdns-browser.html:477, 500, 559, 563, 574 — quote characters in LAN-supplied mDNS names pass through unescaped
- Architecture doc claims adapters are '~150 lines each' (docs/reference/architecture.md:141); actual http.rs is 834 lines — 5.5x the documented size
- Session grace constants: pipe SESSION_GRACE=30s (pipe.rs:13), CLI piped SESSION_GRACE=5s (cli.rs:11); the only documented external IPC consumer is socat NDJSON from containers (CONTAINERS.md:665)
