# Reader: mdns-core

## Summary

The koi-mdns crate (3,069 LOC across 8 files) is the original heart of Koi and it shows: it is the most deliberately designed domain crate, with a genuine three-face boundary (commands/state/events via MdnsCore), a worker-thread isolation pattern for the mdns-sd engine, and a lease lifecycle engine (session/heartbeat/permanent with ALIVE→DRAINING→EXPIRED states) that closely matches its design doc (docs/proposals/implemented/LEASES.md). The registry is the standout: a pure, lock-protected state machine with deterministic time-injected `_at` test variants and 29 fast unit tests covering the full lease lifecycle. The whole crate's 85 tests pass in 0.02s with zero network dependence.

However, the crate's two flagship claims are both partially false in practice. First, the "single import rule" (.agentic/rules/mdns-boundary.md: daemon.rs is the ONLY file that may import mdns-sd) is violated by browse.rs (lines 4, 16, 37) and by daemon.rs's own public-within-crate API (daemon.rs:82 returns mdns_sd::Receiver). The leak is contained inside the crate (nothing escapes the crate boundary), but the project's AI-context docs — its core philosophy artifact — are factually wrong about their own critical rule. Second, and more serious: the facade hands out multiple independent BrowseHandles per service type, but mdns-sd 0.17 keeps exactly ONE querier per type (service_queriers: HashMap<String, Sender>; a second browse "will be updated, i.e. overwritten" per mdns-sd's own source). Consequences: a second concurrent SSE discover of the same type silently kills the first; resolve() internally browses-then-stop_browses a type (daemon.rs:167-179), terminating any concurrent subscriber; BrowseHandle::drop does the same (browse.rs:25); and the daemon-mode MdnsBridge cache (crates/koi/src/integrations.rs:203-233) permanently loses a type's feed when this happens because its `active` set never re-spawns a dead type browse. The clean facade actively hides a shared-mutable-singleton reality, which is the most significant latent bug class in the project's oldest code.

The protocol serde design is a qualified success. The custom Serialize on Response (protocol.rs:149-191) producing wrapped shapes for happy paths and flat shapes for error/event is consistent and well-tested for serialization — but it is one-directional (no Deserialize, so no round-trip tests despite CLAUDE.md mandating "test serde round-trips"), every new variant requires hand-written serializer code, and the wrapped/flat duality is a tacit contract consumers must just know. Meanwhile the PipelineResponse status machinery (ongoing/finished/warning) is dead code in production — every call site in the entire workspace uses `clean()`; ongoing/finished/with_warning appear only in tests and carry #[allow(dead_code)] in koi-common — while the public API reference (.agentic/reference/api-endpoints.md) still documents streaming `status` fields that are never emitted.

As a template for other domains, the crate's shape (lib facade + error + protocol + http + internal modules, co-located constants, thiserror→ErrorCode→HTTP mapping, route-path constants module) has demonstrably propagated: koi-dns/health/proxy/udp/runtime all follow it at comparable sizes (686-1,678 LOC; certmesh is the 18,610 LOC outlier). But the template also exported its warts: dead pipeline status plumbing, Extension-based handler state where the project's own http-adapter rule documents State extractors, and zero test coverage on the facade/daemon layers (lib.rs, daemon.rs, browse.rs, events.rs: 768 LOC, 0 tests — all coverage is on pure logic). Maturity: solid beta for the registry/protocol/HTTP surface; the concurrency semantics around browse multiplexing are still feasibility-test grade.

## Strengths

- Registry lease engine is exemplary: pure state machine with deterministic time injection via _at variants (crates/koi-mdns/src/registry.rs:202-341), 29 lifecycle tests including drain/revive/reconnect/reap edge cases, no sleeps, runs in milliseconds
- Worker-thread isolation of mdns-sd is well executed: MdnsOp enum, bounded sync_channel(256) with try_send backpressure errors, oneshot replies for browse/shutdown, named thread koi-mdns-ops (crates/koi-mdns/src/daemon.rs:21-79); single conversion point resolved_to_record (daemon.rs:252)
- Lease policy design is genuinely elegant: the adapter picks the default (pipe/UDS=session 30s grace, CLI stdin=session 5s, HTTP=heartbeat 90s/30s, explicit permanent) and all three modes are actually exercised in production paths (dispatch.rs:61 session, http.rs:381-395 heartbeat, lib.rs:140-142 permanent), matching the LEASES.md design doc closely
- Exhaustive error-mapping test forces every MdnsError variant to declare its ErrorCode and HTTP status (crates/koi-mdns/src/error.rs:67-127) — adding a variant breaks the build until mapped
- Registration failure rollback: daemon register failure rolls back the registry insert (lib.rs:160-168); reconnect-to-draining semantics absorb container restarts without network-visible interruption
- Route path constants module paths{} with rel() helper is a single source of truth shared by axum routing and the command manifest (crates/koi-mdns/src/http.rs:64-84)
- BrowseHandle implements RAII cleanup (stop_browse on Drop, browse.rs:23-33) so daemon browse resources are never leaked by the handle holder
- 85 tests pass in 0.02s, fully deterministic, zero network dependence; serde shape tests verify the exact wire contract (clean responses have no extra keys, errors are flat, events are flat)
- Crate has publication-grade hygiene: README with badges/keywords/categories, doc comments on all public items, tracing instead of println, no unwrap/expect in production paths (verified by dedicated cleanup commits 97f64e8, e5734a7)

## Concerns

- Boundary rule violated by its own crate: browse.rs imports mdns_sd (crates/koi-mdns/src/browse.rs:4,16,37) and daemon.rs:82 returns mdns_sd::Receiver<ServiceEvent> from its API, contradicting .agentic/rules/mdns-boundary.md:8 ('daemon.rs is the ONLY file that may import') and CLAUDE.md's Never Do list. Contained within the crate, but the AI-context docs are wrong about their flagship rule
- mdns-sd allows only ONE querier per service type (service_queriers: HashMap<String, Sender>, second browse 'will be updated, i.e. overwritten' — mdns-sd 0.17 service_daemon.rs:873,3110), but MdnsCore::browse hands out unlimited 'independent' BrowseHandles. Second concurrent SSE discover of the same type silently kills the first client's stream
- resolve() browses the target type then calls stop_browse on it (crates/koi-mdns/src/daemon.rs:167,173,179), and BrowseHandle::drop does the same (browse.rs:25) — either one terminates ALL concurrent browses/subscribes of that type and clears the mdns-sd cache for it
- MdnsBridge dashboard cache permanently degrades: run_meta_browse's `active` HashSet only inserts (crates/koi/src/integrations.rs:209-228), so when a type browse dies from the stop_browse interference above, that type is never re-browsed for the life of the daemon
- Event model is a passive tap, not a feed: MdnsCore::subscribe() broadcast only receives events while some other consumer actively polls a BrowseHandle (events are re-broadcast inside BrowseHandle::recv, browse.rs:74,84,92); N concurrent handles on one type would duplicate every event N-fold into the channel. The 'Events face' of the domain model is misleading as documented
- Response enum has custom Serialize but no Deserialize (protocol.rs:149-191): wire contract is only half-tested (serialization only), violating the project's own 'test serde round-trips' rule; KoiClient must hand-parse responses; each new variant requires manual serializer code and tacit knowledge of the wrapped-vs-flat convention
- PipelineResponse status/warning machinery is dead in production — only clean() is called anywhere in the workspace; ongoing()/finished()/with_warning() exist only in tests with #[allow(dead_code)] (koi-common/src/pipeline.rs:40-60) — while .agentic/reference/api-endpoints.md still documents streaming 'status: ongoing/finished' responses that are never emitted
- Removed events carry the raw fullname in ServiceRecord.name with empty service_type (browse.rs:87-91), forcing downstream consumers to re-parse (integrations.rs extract_service_type/extract_instance_name); ServiceRecord.name is semantically overloaded three ways (instance name, type name for meta-query, fullname for removals)
- Duplicate ALIVE registrations of the same name+type are permitted (registry.rs test insert_does_not_reconnect_alive_entry:477-497); both map to the same mDNS fullname, so unregistering one sends goodbye packets that kill the other's advertisement while its registry entry still claims alive — silent registry/network divergence
- Zero test coverage on the orchestration layer: lib.rs (337 LOC, reaper task, rollback logic, reconnect re-register), daemon.rs (326 LOC, worker loop, resolve), browse.rs (105 LOC) have no tests and no mockable seam (MdnsDaemon is a concrete struct, not a trait)
- Reconnected-with-changed-payload re-register failure only logs a warning (lib.rs:181-186) — registry says alive, network advertisement is gone
- resolve() splits instance names with splitn(2, '.') (daemon.rs:147) — DNS-SD instance names may legally contain dots, causing wrong type extraction
- set_requires_probe(false) (daemon.rs:131) deliberately skips RFC 6762 conflict probing — justified for stale-self-records, but Koi will never detect genuine name collisions with other hosts
- Loopback-IP substitution hack (daemon.rs:279-288) silently rewrites loopback addresses to the first non-loopback IPv4 from lan_ip() — helpful for containers, wrong for genuinely loopback-bound services and arbitrary on multi-homed machines
- http-adapter rule doc (.agentic/rules/http-adapter.md) documents State(core) extractors; the implementation uses Extension(core) (http.rs:104,112) — another agentic-docs drift
- browse_handler and events_handler in http.rs are ~95% duplicated 30-line SSE stream loops (http.rs:126-154 vs 227-256), differing only in the event→pipeline mapping function — the project factored this pattern on the CLI side (run_streaming) but not here

## Maturity signals

- Beta-grade for registry/protocol/HTTP: 85 deterministic tests passing in 0.02s, exhaustive error mapping, design doc (LEASES.md) faithfully implemented, dedicated safety-cleanup commits removing unwrap/expect (git: 97f64e8, e5734a7)
- Publication trappings present: crates.io/docs.rs badges, keywords, categories, per-crate README — signals intent to ship as a real library
- Feasibility-test grade for browse concurrency: the single-querier-per-type reality of mdns-sd is unmodeled, producing silent cross-request stream kills and permanent dashboard-cache degradation — these are bugs you only find under concurrent multi-client use, i.e. nobody has run this hard yet
- Agentic context docs have drifted from code in at least three places (single-import rule, pipeline streaming status contract, State vs Extension handler pattern) — documentation discipline is aspirational rather than enforced
- Git history shows healthy iteration: 13 commits on the crate over ~6 weeks (2026-02-12 to 2026-03-25), including a v0.1→v0.2 lease rewrite, dead-code removal (a708655), and CI/clippy hygiene passes
- Test coverage is skewed: 100% of tests target pure logic (registry, serde, helpers); the 768 LOC of actual mDNS interaction (lib.rs facade, daemon worker, browse handle) has zero coverage and no mock seam — typical of a project that has tested what is easy rather than what is risky

## Shed or simplify

- Restore the single-import rule honestly: have MdnsDaemon::browse return a crate-owned receiver wrapper (or move BrowseHandle into daemon.rs) so mdns_sd types never leave daemon.rs; alternatively amend .agentic/rules/mdns-boundary.md to say 'daemon.rs + browse.rs'. Either fix is small; the current state is the worst option (rule and reality disagree)
- Add browse multiplexing inside MdnsDaemon: one real mdns-sd browse per type, fanned out to N handles with refcounted stop_browse-on-last-drop. This single change fixes the concurrent-SSE kill, the resolve() interference, and the MdnsBridge permanent-degradation bug, and makes the facade's promise true. Highest-value structural fix in the crate
- Delete the PipelineResponse status/warning machinery (PipelineStatus enum, ongoing/finished/with_warning, koi-common/src/pipeline.rs:4-64) or actually emit finished on stream close — it is workspace-wide dead code that contradicts the published API docs; if deleted, PipelineResponse collapses to nearly nothing and could itself be questioned
- Factor the duplicated SSE stream loop in http.rs (browse_handler:126-154, events_handler:227-256) into one helper taking the event→pipeline mapping fn — mirrors the run_streaming dedup already done on the CLI side
- Normalize the Removed event at the boundary: parse fullname into instance name + service type once in daemon.rs (which already owns conversion), delete extract_service_type/extract_instance_name from integrations.rs, and remove the ServiceRecord.name overloading
- Decide duplicate-registration semantics: either reject a second ALIVE registration of the same (name, type) with ErrorCode::Conflict (the error code already exists and is unused here) or document that duplicates share one network advertisement; current silent divergence is a trap
- Merge events.rs (10 LOC) into lib.rs or protocol.rs — a file-per-concept is good, but a 10-line file whose type is re-exported from lib.rs anyway is ceremony
- Consider collapsing Response::Found and Response::Resolved (both wrap ServiceRecord; browse_event_to_pipeline already collapses them at protocol.rs:203) — one less variant in the hand-written serializer
- Give Response a Deserialize impl (or restructure as two enums: wrapped happy-path with standard externally-tagged serde + flat error/event) so round-trip tests become possible and KoiClient stops hand-parsing — the custom Serialize exists mainly to serve the two flat variants
