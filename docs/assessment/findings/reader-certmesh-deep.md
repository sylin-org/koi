# Reader: certmesh-deep

## Summary

Certmesh is Koi's private-PKI pillar: a passphrase/envelope-encrypted ECDSA P-256 CA (crates/koi-certmesh/src/ca.rs) that mints 30-day member certs, enrolls machines over HTTP with TOTP (or nominally FIDO2) verification plus rate limiting and scope policy (enrollment.rs), persists a roster (roster.rs), auto-renews via an hourly loop with validated no-shell reload hooks (lifecycle.rs), supports primary/standby failover with CA-signed roster manifests, mDNS absence detection, and X25519-DH-encrypted CA key transfer (failover.rs), encrypted backup/restore bundles (backup.rs), roster-level revocation, an append-only text audit log (audit.rs), and a 'compliance' summary that is essentially a policy echo plus an audit line count (http.rs:868-896). koi-crypto supplies the LUKS-inspired unlock-slot table (passphrase/auto-unlock/TOTP/FIDO2 slots, unlock_slots.rs), Argon2id+AES-GCM key encryption, keyring-based machine binding mislabeled 'tpm' (tpm.rs), a vault, TOTP, and X25519 agreement. koi-truststore is a clean 306-line OS trust-store installer.

The headline statistic is misleading: wordlist.rs is 7,784 lines (41.8%) of the 18,610-line crate — it's the EFF wordlist embedded as a Rust constant. Real certmesh code is ~10.8k lines, of which a large fraction is tests (263 test functions; lib.rs alone is ~980 lines of tests out of 2,435). Engineering hygiene is genuinely high: thiserror enums mapped to wire ErrorCodes, spawn_blocking around all file I/O, deliberate lock-drop before blocking, Zeroizing on secrets, serde back-compat tests, and a visible multi-round security-review history in git plus a self-aware consolidation ADR (docs/adr/012). This is not slop.

The strategic problem is scope, not craft. The core loop (create CA → join with TOTP → trusted certs → auto-renew → truststore install) is a killer homelab feature and fits Koi's philosophy. But layered on top is an enterprise PKI: three trust profiles plus custom, operator-approval workflows with timeouts, enrollment windows with deadlines, domain+CIDR scope policies, signed roster manifests, automatic failover with lexicographic tiebreakers, FIDO2 enrollment AND FIDO2/TOTP unlock slots, compliance summaries, and versioned encrypted backups. Several of these are demonstrably speculative: FIDO2 input is hard-bailed in the CLI (ceremony_cli.rs:277-279), unlock_with_totp/unlock_with_fido2 have zero callers outside the crate, the FIDO2 slot is marked insecure in a TODO (unlock_slots.rs:95-97), and roster.rs:62 references 'Moss' and 'Rake' — vocabulary from a sibling Sylin ecosystem, suggesting certmesh was designed for a bigger product family than Koi. Counting binary-side wiring (commands/certmesh.rs 1,335 lines, ceremony_cli.rs 564, ~470 lines of background-task orchestration in main.rs:917-1290), the certmesh pillar consumes roughly 45% of the 55k-line codebase.

Coupling is well-disciplined — certmesh depends only on koi-common/koi-crypto/koi-truststore, with cross-domain wiring confined to the binary via bridge traits — but the crate violates its own house rules internally: CA creation logic lives exclusively in a 230-line HTTP handler (http.rs:252-482, explicitly documented at lib.rs:286-289), contradicting the 'handlers delegate to domain methods' facade rule; eval_init in pond_ceremony.rs is a single ~755-line function (lines 79-835); HOOK_FORBIDDEN is defined twice (lib.rs:576, http.rs:178) against the 'never duplicate constants' rule; and a clone-roster/spawn_blocking/save boilerplate block is copy-pasted ~12 times through lib.rs.

## Strengths

- Exceptional test density for a hobby project: 263 #[test]/#[tokio::test] functions in koi-certmesh, ~99 in koi-crypto, 6 in koi-truststore; tests cover error paths, serde backward compatibility (roster.rs:410-435, 600-614), DH round-trips, tampered-manifest rejection, and rate-limit lockout
- Consistent async hygiene: every roster/auth/cert file write goes through tokio::task::spawn_blocking, and locks are explicitly dropped before blocking work (e.g., lib.rs:489, lib.rs:1190-1203 releases the roster lock before executing reload hooks)
- Real security engineering, not theater: envelope encryption with master key + slot table (koi-crypto/src/unlock_slots.rs), Argon2id+AES-256-GCM at rest, X25519 ephemeral DH for CA key transfer so passphrases never cross the wire (failover.rs:40-119), Zeroizing wrappers, reload hooks validated against shell metacharacters and executed without a shell (lifecycle.rs:60-69), CN-based authorization on the mTLS inter-node port (http.rs:163-174)
- Self-aware governance: ADR-012 documents an executed consolidation with 87 catalogued findings, dead-code inventory with dispositions, and honest DEFERRED markers for FIDO2; git history shows 4+ rounds of security-review fix commits
- Clean dependency direction: certmesh depends only on koi-common/koi-crypto/koi-truststore; integration with health/dashboard goes through CertmeshSnapshot bridge traits implemented in the binary crate — verified zero cross-domain imports (ADR-012 Block 4 'SKIP — verified')
- koi-truststore is a model small crate: 306 lines total, strict name validation (lib.rs:37-62), per-platform modules of 34-46 lines each, errors designed to be non-fatal
- Failure-tolerant design choices: renew_all_due returns per-member Results without short-circuiting (lib.rs:1123-1162); machine-binding mismatch warns instead of bricking the CA after OS reinstalls (koi-crypto/src/keys.rs:198-207); truststore install is best-effort
- Pluggable auth adapter abstraction (koi-crypto/src/auth.rs) is small, stateless, and well-documented with a clear extension recipe

## Concerns

- wordlist.rs is 7,784 lines — 41.8% of the crate's claimed 18.6k LOC is a static EFF wordlist pasted as a Rust const array; it distorts every size metric and could be an include_str! of a text asset or the eff-wordlist crate
- eval_init in pond_ceremony.rs is a single ~755-line function (lines 79-835) implementing the init ceremony as one imperative cascade — flagrantly violates the project's own <50-line-function and file-organization rules; eval_join/invite/unlock are tiny by comparison
- CA creation logic lives exclusively in the 230-line HTTP create_handler (http.rs:252-482), with lib.rs:286-289 explicitly justifying the absence of a CertmeshCore::create() method — this inverts the project's own Domain Facade Pattern ('HTTP handlers delegate to domain methods') for the single most important operation in the domain
- FIDO2 is a dead-end feature shipped in three layers: auth adapter (koi-crypto/src/auth.rs:283+), unlock slot marked 'experimental — credential_id-derived storage key is insecure' (unlock_slots.rs:95-97), and protocol variants — but the CLI hard-bails ('FIDO2 hardware key input is not yet supported', crates/koi/src/commands/ceremony_cli.rs:278) and rotate_auth rejects it (lib.rs:948-955); unlock_with_fido2 AND unlock_with_totp (lib.rs:670-720) have zero callers outside the crate
- Revocation is roster-layer only: no CRL/OCSP distribution, so a revoked member's certificate remains valid to every TLS verifier that trusts the installed CA until its 30-day expiry — revocation only blocks koi-mediated renewal/health/enrollment; this gap is not surfaced to users
- No CSR flow: the CA generates member private keys server-side and ships key_pem over the wire on enrollment and renewal pushes (protocol.rs RenewRequest, ca.rs:288) — acceptable for a LAN tool but a real PKI design shortcut that deserves explicit documentation
- Copy-paste roster persistence: the clone-roster → spawn_blocking → save_roster → map_err block appears ~12 times across lib.rs (e.g., 376-387, 514-524, 595-601, 619-625, 808-814, 839-845, 879-885, 1092-1098, 1146-1158, 1238-1248, 1283-1289, 1345-1351, 1372-1378, 1404-1411) — a single persist helper would remove ~120 lines and one class of bug
- CertmeshState holds 7 separate tokio::Mutex fields (lib.rs:66-77) acquired in varying orders (enroll takes ca→roster→auth→challenge→rate_limiter at lib.rs:333-350); correctness rests on convention with no documented lock ordering
- HOOK_FORBIDDEN constant duplicated in lib.rs:576-579 and http.rs:178-181 — direct violation of the project's 'never define the same constant in multiple files' rule; the two validators also differ (HTTP adds absolute-path checks the domain method lacks)
- CertmeshError::Internal(String) is a catch-all that erases semantics: member-not-found in set_reload_hook maps to Internal/500-ish instead of NotFound (lib.rs:592, http.rs:238-241); invalid hostname maps to Internal (lib.rs:321)
- build_ca_state_from_der (ca.rs:243-277) re-creates the self-signed CA certificate with fresh not_before/not_after on every load instead of parsing the on-disk cert for use as rcgen issuer — works only because issuer DN and SKI derive from the key; subtle and fragile against rcgen changes
- Machine-binding sealing logic exists twice with the same 'koi-certmesh-ca' label: ca.rs:141-149 (create path) and koi-crypto/keys.rs:174-182 (encrypt_key) — overlapping responsibilities across crates
- main.rs carries ~470 lines of certmesh orchestration (spawn_certmesh_background_tasks at main.rs:917-1290: roster sync loop, failover watch, renewal loop, approval prompt) — straining the documented 'main.rs contains zero business logic' rule
- Tests share the real default data dir via koi_common::test::ensure_data_dir (ca.rs:362, lib.rs:1604) rather than tempdirs — capability_status_uninitialised even branches on whether the dev machine has CA files (lib.rs:2349-2371), an admitted test-environment leak
- tpm.rs is not TPM — it is the keyring crate (Credential Manager/Keychain/Secret Service); the module name oversells the security property (acknowledged in its own doc comment)
- Audit log is plain append-only text with no tamper-evidence (no hash chaining, no permissions hardening mentioned in audit.rs) — fine as a journal, but it underwrites the 'compliance' feature's credibility

## Maturity signals

- Beta-grade core path: CA create/join/status/unlock/renew is feature-complete, heavily unit-tested (807+ tests repo-wide claimed in ADR-012), security-reviewed across 4+ commit rounds (git log: 'fix(security): resolve round-4 review findings', 'security: migrate secret material to Zeroizing<T>'), with legacy-format auto-migration (ca.rs:194-216) and roster serde back-compat — evidence of real upgrade-path thinking
- Alpha-or-below features shipped alongside: FIDO2 (non-functional end-to-end, CLI bails at ceremony_cli.rs:278, slot marked insecure), TOTP-unlock slots (no callers), automatic failover (requires multi-node daemon deployment; ~684-line failover.rs + main.rs:1194+ watch loop — no integration test can exercise this), compliance endpoint (~30 lines echoing policy + counting audit lines)
- 13 ADRs including an executed consolidation roadmap (ADR-012, 'Status: Accepted (executed 2026-03-25)') with blocks marked DONE/SKIP/DEFERRED and commit hashes — unusually mature process for a feasibility-test project
- 49 commits touching crates/koi-certmesh, all since early 2026 — fast-moving, recently churned subsystem, not settled code
- Vocabulary drift reveals transplanted ambition: 'pond'/'garden' metaphors, MemberRole::Client documented as 'Non-Moss client (e.g. Rake on a workstation)' (roster.rs:62), unlock_slots.rs:69 'Moss writes/reads it' — certmesh's requirements were drawn from a broader Sylin product family, not from Koi's own user base
- A defensive publication for the envelope-encryption scheme exists (docs/prior-art/defensive_pub_heterogeneous-envelope-encryption.md) — signals the authors consider koi-crypto's slot design novel and potentially standalone
- Phase-numbered comments throughout (Phase 2+/3/4/5/8 in lib.rs, roster.rs, error.rs) show the subsystem was built to a staged spec rather than grown from usage

## Shed or simplify

- Replace wordlist.rs (7,784 lines) with include_str!('eff_large_wordlist.txt') parsed once via LazyLock, or the existing eff-wordlist crate — removes 42% of the crate's LOC at zero functional cost (only consumer is entropy.rs:16)
- Remove or feature-gate all FIDO2 code until a working authenticator integration exists: Fido2Adapter + Fido2Credential (koi-crypto/auth.rs), UnlockSlot::Fido2 + unwrap_with_fido2 (unlock_slots.rs), unlock_with_fido2 (lib.rs:701-720), protocol Fido2 variants, and possibly the p256 dependency — ADR-012 already defers it (7.2/7.5) and marks the slot insecure; shipping unreachable crypto paths is pure liability (~500+ lines)
- Delete the compliance endpoint and CLI command (http.rs:868-896, commands/certmesh.rs) — it returns the policy block (already in /status) plus a line count of the audit log (already in /log); ~100 lines plus protocol types for no information
- Demote automatic failover to manual promotion: keep prepare/accept_promotion (the DH transfer is good), drop the mDNS absence-watch loop, tiebreaker, and roster-sync background tasks (failover.rs detection half + main.rs:917-1290 wiring + ROSTER/HEALTH inter-node plumbing) — for a LAN tool with 30-day certs, a dead CA means 'renewals pause for days', not an outage; this is the largest block of plausibly-never-exercised machinery (~800+ lines across crates)
- Move create_handler's orchestration (http.rs:252-482) into CertmeshCore::create(), honoring the project's own facade rule and making CA creation testable without axum; the stated divergence-avoidance rationale (lib.rs:286-289) argues for one domain method, not for HTTP-only logic
- Extract a single CertmeshCore::persist_roster() helper to replace the ~12 duplicated clone/spawn_blocking/save blocks in lib.rs (~120 lines saved, one consistency point for the save-failure policy which currently silently warns in some paths and errors in others)
- Split pond_ceremony.rs eval_init (755 lines) into per-step functions (profile, policy overrides, entropy, passphrase, unlock-method, TOTP verification) — the file already does this for join/invite/unlock
- Unify the duplicated HOOK_FORBIDDEN validation into one koi-certmesh function used by both lib.rs:584 and http.rs:182 (and reconcile the absolute-path check that only the HTTP layer performs)
- Consider extracting koi-crypto (+ possibly certmesh+truststore) into a standalone repo/crate: it has zero koi-common entanglement beyond encoding helpers, a defensive publication already positions the slot-table design as independent work, and the pond/Moss/Rake vocabulary shows it serves a wider ecosystem — Koi would then consume a versioned dependency instead of hosting a third of its LOC for one pillar
- Simplify trust profiles: cert_lifetime_days() returns 30 for all profiles (profiles.rs:39-41) and the 'custom' ceremony path reduces to two booleans (enrollment_open, requires_approval) — exposing those two flags directly would remove a layer of indirection that currently fans into ceremony prompts, roster metadata fallbacks (requires_approval Option dance, roster.rs:184-189), and approval plumbing
