# Reader: embedded-dx

## Summary

This lens covered Koi's three "consumer-facing" crates: koi-embedded (embed the whole toolkit in a Rust app), koi-client (blocking HTTP client for the daemon), and command-surface (a generic terminal help-rendering mini-framework). The headline finding is that koi-embedded is a second orchestrator that openly duplicates the binary crate's wiring — lib.rs:1066 literally says "Duplicated from the binary crate's integrations.rs". The five integration bridges (~210 lines), the dashboard SSE event forwarder (~115 lines, near-verbatim copy of adapters/dashboard.rs:349-460), the capability-status assembly (which now exists in four near-identical copies across koi/adapters/http.rs, koi-embedded/http.rs, and koi-embedded/lib.rs build_embedded_snapshot), and a 505-line second HTTP adapter all mirror the binary. koi-embedded even reimplements certmesh auto-unlock with a hardcoded "auto-unlock-key" filename (lib.rs:836) despite koi-certmesh exposing auto_unlock_key_path() and try_auto_unlock() — three sources of truth for one mechanism. The 5.8k lines decompose as: ~3.9k src (of which ~800 are in-file unit tests, heavily skewed toward trivial enum-construction assertions), 936 test lines, and a 1,075-line hand-rolled PASS/FAIL "example" that is really an integration test wearing the wrong hat. The builder API itself is decent Rust (fluent, sensible defaults, ServiceMode::Auto/EmbeddedOnly/ClientOnly), but has naming inconsistencies (.mdns(bool) vs .dns_enabled(bool) vs .runtime(kind)), a side-effecting builder method (ensure_firewall_rules creates Windows Firewall rules mid-chain), leaks hickory_proto types in DnsHandle::lookup, silently degrades in remote mode (dead event streams, errors swallowed into defaults), and ships a README whose example does not compile (.mode() doesn't exist, build() isn't async).

koi-client (728 lines) is the healthiest of the three: clear job, typed errors, a tested SSE iterator, token auth, breadcrumb discovery, and two real consumers (the CLI's client mode with 36+ call sites, and koi-embedded's remote backend). Blocking ureq is a defensible choice for a CLI client — the cost shows up as spawn_blocking boilerplate throughout koi-embedded/handle.rs. Its two real flaws: only mDNS operations return typed responses while DNS/health/proxy/UDP/certmesh return raw serde_json::Value (forcing koi-embedded to hand-parse JSON with fragile extractors in handle.rs:774-846), and its dependency tree contradicts its own "no async runtime dependency" claim — depending on koi-mdns and koi-health for a handful of types drags mdns-sd, axum, tokio, and utoipa into every consumer's build graph.

command-surface (1,043 lines, publish=false, zero tests, exactly one consumer) is the clearest yak-shave. It is a generic Category/Tag/Scope/Glyph trait framework with terminal-capability detection, three catalog renderers, a detail view, and two writers — built so that `koi` help output can show colored glyphs and badges. The output it produces is genuinely a DX pillar of the project, but the generic machinery is unjustified: write_summary_catalog (96 lines), by_tag, by_scope, and the entire Confirmation gate API are dead code; the manifest declares Confirmation::TypeToken on two commands (surface.rs:1371,1920) while actual confirmation is hand-rolled separately in commands/factory_reset.rs:36 and commands/certmesh.rs:1047 — decorative metadata with duplicated enforcement. Project docs also misstate its dependencies (CONTEXT.md says crossterm; it actually uses console/is-terminal/supports-color/terminal_size). Verdict on "earning their place": koi-client yes (with a types diet); koi-embedded's concept earns its place — embedding is arguably the differentiator of the single-binary philosophy — but the implementation should either share a composition layer with the binary or become the orchestrator the binary wraps; command-surface should be folded into the binary as a concrete module, deleting the generics and dead paths.

## Strengths

- koi-embedded's builder surface is idiomatic fluent Rust with sensible safe defaults (mdns+dns on, everything else opt-in; KoiConfig::default at config.rs:72-98) and a clean three-mode ServiceMode (Auto/EmbeddedOnly/ClientOnly) that lets the same code run embedded or against a shared daemon
- The handle pattern (KoiHandle -> MdnsHandle/DnsHandle/HealthHandle/... with enum-dispatched Embedded/Remote backends, handle.rs:21-34) gives consumers one API regardless of where Koi runs — a genuinely good embedding DX idea
- Disabled capabilities fail loudly and consistently via KoiError::DisabledCapability rather than panicking or silently no-op'ing (handle.rs:103-208), and tests verify this (tests/udp.rs udp_disabled_returns_error)
- koi-client is small, focused, dependency-light at runtime (blocking ureq, no tokio needed by consumers at runtime), with typed ClientError, DAT token auth on every verb (lib.rs:88-125), breadcrumb auto-discovery (from_breadcrumb, lib.rs:82-85), and a simple well-unit-tested SSE line parser (lib.rs:511-537, 13 tests)
- Graceful shutdown is taken seriously in koi-embedded: KoiHandle::shutdown cancels tasks, stops runtimes in order, withdraws the mDNS HTTP announcement before mdns core shutdown (handle.rs:210-246)
- Real integration tests exist for the hard paths: tests/udp.rs (787 lines) exercises bind/send/recv/heartbeat/lease-expiry end-to-end through the embedded API
- command-surface's rendered output (capability-aware help catalogs, ? detail views with examples + HTTP API equivalents, NO_COLOR/TERM=dumb/non-TTY degradation in profile.rs:70-107) is a genuine CLI DX differentiator, whatever one thinks of the crate boundary
- koi-embedded's start() degrades gracefully when optional backends are missing (runtime backend unavailable -> warn and continue, lib.rs:397-407)

## Concerns

- Two-orchestrators drift is real and admitted: koi-embedded/src/lib.rs:1066-1068 comments 'Duplicated from the binary crate's integrations.rs'. Five bridges + run_meta_browse/run_type_browse (lib.rs:1070-1230, 1629-1673) are near-verbatim copies of crates/koi/src/integrations.rs (284 lines)
- Dashboard SSE event forwarder duplicated: koi-embedded/lib.rs:455-558 vs crates/koi/src/adapters/dashboard.rs:349-460 — a ~115-line select! block over six broadcast channels that must be updated in two places whenever any domain adds an event variant
- Capability-status assembly exists 4+ times: koi/adapters/http.rs:510-610, koi-embedded/src/http.rs:270-379 (status_handler), koi-embedded/src/lib.rs:958-1064 (build_embedded_snapshot), plus the binary dashboard snapshot — same disabled/stopped/running ladder per domain in each
- koi-embedded reimplements certmesh auto-unlock with a hardcoded path string resolved_data_dir.join("auto-unlock-key") (lib.rs:836) despite koi-certmesh exposing AUTO_UNLOCK_KEY_FILENAME, auto_unlock_key_path() and try_auto_unlock() (koi-certmesh/src/lib.rs:761) — violates the project's own 'never duplicate constants' rule and will drift
- koi-embedded README example does not compile: shows .mode(ServiceMode::Standalone) (method is service_mode, variant doesn't exist), build().await (build is sync; start() is the async step), and koi.mdns().register(...).await (mdns() returns Result; register is sync). koi-client README similarly shows client.status() which doesn't exist (it's unified_status())
- KoiEmbedded::start() is a ~560-line monolith (lib.rs:247-807) containing six near-identical event-forwarder spawn blocks (lib.rs:668-791) — violates the project's own <50-line function and 800-line file guidance (lib.rs is 1,673 lines)
- Remote mode silently degrades: MdnsHandle::subscribe returns a dead receiver (handle.rs:370-378), KoiHandle::events()/.events(handler) never fire (no forwarders spawned in new_remote path, lib.rs:262), DnsHandle::stop/HealthHandle::status/ProxyHandle::entries swallow errors into false/empty defaults (handle.rs:481-494, 548-564, 702-715) — violates 'never silently swallow errors' and gives embedders no failure signal
- ServiceMode::Auto is the default: an embedding app silently becomes a client of whatever daemon answers on localhost:5641, and all builder capability config (dns ports, data_dir, etc.) is silently ignored in that path (lib.rs:252-276) — surprising behavior change based on host machine state
- Builder API naming is inconsistent: .mdns(bool)/.health(bool)/.proxy(bool)/.udp(bool)/.http(bool) vs .dns_enabled(bool) (because .dns() is taken by a config closure) vs .runtime(kind) which enables implicitly and takes a backend enum (lib.rs:93-162)
- ensure_firewall_rules() performs system mutation (creates Windows Firewall rules) inside the builder chain (lib.rs:216-225) — a side effect where consumers expect pure configuration; its doc comments hardcode consumer names ('Moss discovery UDP', 'Zen Garden') revealing the API was shaped for one private downstream app
- hickory_proto::rr::RecordType leaks into the public embedded API (DnsHandle::lookup, handle.rs:421-424) without re-export, forcing consumers to add hickory-proto with a matching version to their own Cargo.toml
- koi-client's typed/untyped split: mDNS ops return typed structs but all DNS/health/proxy/UDP/certmesh ops return raw serde_json::Value (lib.rs:218-482), pushing fragile hand-rolled JSON extraction into koi-embedded (parse_dns_lookup/parse_dns_entries/parse_proxy_entries/extract_capability_status, handle.rs:774-846) — a stringly-typed seam that drifts independently from domain http response types
- koi-client's dependency tree contradicts its stated rationale ('no async runtime dependency on the client path', lib.rs:3): depending on koi-mdns and koi-health for a few protocol types pulls mdns-sd, axum, tokio, and utoipa into every consumer's build graph (confirmed via cargo tree)
- record_type_str silently maps any unknown RecordType to "A" (koi-client/src/lib.rs:568-575) — silent fallback on a query parameter
- command-surface has zero tests, publish=false, one consumer (crates/koi/src/surface.rs), and significant dead API: write_summary_catalog (default.rs:89-185), CommandManifest::by_tag/by_scope, requires_confirmation()/gate() — none used anywhere in the workspace
- Confirmation metadata is decorative: surface.rs declares Confirmation::TypeToken on two commands (lines 1371, 1920) but actual prompts are hand-rolled in commands/factory_reset.rs:36-39 and commands/certmesh.rs:1047 — the manifest claims an enforcement role it doesn't have
- Docs drift in project context: CONTEXT.md crate table says command-surface depends on crossterm; it actually uses console/is-terminal/supports-color/terminal_size (command-surface/Cargo.toml). CONTEXT.md also omits koi-embedded's hickory-proto/reqwest/netdev/if-addrs deps
- Test calories spent in the wrong place: events.rs is 203/248 lines of tests asserting that enum variants can be constructed and Debug doesn't panic; meanwhile the 560-line start() orchestration has no direct test, and the real integration suite lives in a 1,075-line examples/embedded-integration.rs with a hand-rolled [PASS]/[FAIL] harness instead of cargo test
- examples/embedded-basic.rs (19 lines) is misleading: main returns immediately after spawning the event loop, so it prints nothing when run
- koi-embedded/src/lib.rs has no crate-level //! rustdoc at all, and most builder/handle methods are undocumented — weak docs.rs story for a crate with published-crate badges in its README

## Maturity signals

- Alpha overall for these three crates: repo is 176 commits total; koi-embedded history starts ~Feb 2026 and shows rapid feature accretion (vault, runtime adapter, dashboard, firewall) with fix-up commits ('eliminate unwrap/expect in production paths', 'embedded crate hardening') following features
- Version string 0.2.202603241449 (date-stamped patch) signals pre-release, not semver discipline — no published crates.io history despite badges and metadata
- Both consumer-facing READMEs contain non-compiling/nonexistent-API examples (koi-embedded README .mode()/build().await; koi-client README client.status()) — documentation written aspirationally and never verified, a prototype/feasibility tell
- command-surface is publish=false with zero tests and one consumer — an internal experiment, not a library
- Production-grade signals exist in pockets: deliberate error taxonomy (KoiError with #[from] per domain), graceful shutdown ordering, lease/heartbeat tests for UDP (787 lines), poisoned-lock recovery (unwrap_or_else(|e| e.into_inner()) in lib.rs:1145), CI-hardening commits (b900a3c macOS keychain probe skip)
- Test distribution is feasibility-stage: high-count trivial unit tests (enum construction, Debug-doesn't-panic) inflate coverage while the orchestration core (start(), 560 lines) and the remote-mode handle paths have no automated tests; the real end-to-end suite is an example binary with a hand-rolled harness, not part of cargo test
- The duplication comment at lib.rs:1066 ('Duplicated from the binary crate's integrations.rs because...') shows the team knows about the two-orchestrator problem and consciously deferred it — a feasibility-test tradeoff, not an accident
- Consumer-specific doc references ('Moss discovery UDP', 'Zen Garden' in lib.rs:198-215) indicate koi-embedded is being grown against one private downstream app rather than a generalized API design

## Shed or simplify

- Merge the two orchestrators: extract the duplicated composition layer — integration bridges, meta/type browse cache, dashboard SSE forwarder, capability-status assembly, HTTP adapter route mounting — into one shared crate (koi-compose or grow koi-common's existing dashboard/browser modules), consumed by both crates/koi and koi-embedded. This deletes ~600-700 duplicated lines immediately (lib.rs:1066-1230+1629-1673 vs integrations.rs; lib.rs:455-558 vs dashboard.rs:349-460; four copies of status assembly) and eliminates the structural drift risk. The more radical and cleaner option: make the koi binary a thin CLI shell over koi-embedded — one orchestrator, the embedded crate becomes the product
- Fold command-surface into crates/koi as a concrete module (e.g. src/help/): drop the Category/Tag/Scope generic trait machinery (one consumer, concrete enums already exist in surface.rs:301-409), delete dead API (write_summary_catalog 96 lines, by_tag, by_scope, gate machinery), and either wire Confirmation into actual dispatch or delete it and keep the hand-rolled prompts in factory_reset.rs/certmesh.rs as the single mechanism. Net deletion: a whole crate and ~300+ lines of unused generics
- Put koi-client on a types diet: move the protocol/response types it needs (RegisterPayload, RegistrationResult, ServiceCheckKind, etc.) into koi-common (which it already depends on) or accept string constants, then drop the koi-mdns and koi-health dependencies. This removes mdns-sd/axum/tokio/utoipa from every client consumer's build graph and makes the 'lightweight blocking client' claim true
- Give koi-client typed responses for DNS/health/proxy/UDP (the response structs already exist in each domain's http module per the reference docs) and delete koi-embedded's hand-rolled JSON parsers (handle.rs:774-846, ~75 lines) — one wire contract instead of three
- Replace koi-embedded's hand-rolled auto-unlock (lib.rs:831-877, ~47 lines) with koi-certmesh's existing CertmeshPaths::auto_unlock_key_path() + try_auto_unlock() — the domain crate already owns this logic
- Collapse the six near-identical event-forwarder spawn blocks in start() (lib.rs:668-791, ~125 lines) into one generic helper: fn forward<E>(rx: broadcast::Receiver<E>, map: impl Fn(E)->Option<KoiEvent>, ...) — same for the binary crate's copy
- Convert examples/embedded-integration.rs (1,075 lines, hand-rolled PASS/FAIL harness) into ordinary #[tokio::test] integration tests under tests/, and fix or delete examples/embedded-basic.rs (currently exits before printing any event)
- Prune low-value unit tests: events.rs's 203 lines of variant-construction/Debug tests and similar blocks in config.rs/lib.rs assert that Rust works, not that Koi works; replace with tests for start() composition, remote-mode handles, and ServiceMode::Auto fallback
- Unify builder naming before any publish: either .dns()/.mdns()/.health() all take bools and config moves to .dns_config(closure), or all capabilities get _enabled suffixes; make .runtime() consistent with the rest; consider moving ensure_firewall_rules() off the builder into an explicit post-build call or a KoiHandle method so the builder stays side-effect-free
- Decide remote-mode semantics explicitly: either implement event streaming over the daemon's SSE feeds so .events()/subscribe() work in ClientOnly/Auto-remote mode, or return KoiError::DisabledCapability from them in remote mode — the current silent dead channels are the worst option
