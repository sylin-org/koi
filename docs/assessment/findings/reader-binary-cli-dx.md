# Reader: binary-cli-dx

## Summary

The koi binary crate (14,077 LOC, 30 files, package `koi-net` v0.2.202603241449) is the wiring and DX layer for an 8-domain daemon. The CLI's surface design is genuinely strong: a consistent `koi <domain> <cmd>` moniker structure, bare `koi` printing live daemon status plus a glyph-rendered command catalog, bare `koi <domain>` printing curated per-category examples, and a novel `koi <command>?` help-query syntax (main.rs:44-61). Terminal rendering degrades properly (NO_COLOR, TERM=dumb, non-tty → ASCII glyphs). Platform service management is unusually thorough — Windows SCM upgrade flow with deferred-delete polling, recovery policies, and firewall rule lifecycle; systemd Type=notify; launchd. The clap surface itself (cli.rs, ~717 lines of definitions + 714 lines of tests) is maintainable and well-tested at the parse level.

The problems cluster around duplication and drift, and they are not cosmetic. First, the project maintains three parallel command descriptions — clap definitions, the hand-written 1,931-line surface.rs manifest (54 CommandDef entries, zero tests), and docs — and the manifest has measurably drifted: it registers `certmesh rotate-totp` while the real command is `rotate-auth` (so `koi certmesh rotate-auth?` fails), and its long descriptions advertise flags that do not exist (`--totp`, `--exec`, `--cidr`, `--process`, `--include-logs`) plus DNS record types the parser rejects. The manifest's Confirmation machinery (`gate()`, `requires_confirmation()`) is dead code; actual confirmation prompts are hand-rolled twice. Second, windows.rs `run_service` duplicates ~400 lines of main.rs `daemon_mode()` wiring nearly verbatim and has drifted behaviorally: the Windows service never spawns the runtime orchestrator or the certmesh renewal/roster-sync/heartbeat/failover loops, so `koi install` on Windows yields a materially weaker daemon than `koi --daemon`. Third, main.rs (1,545 lines) violates the project's own "zero business logic" rule — lines 917-1292 contain a full certmesh failover state machine (promote/demote, mDNS CA announcement) living in the binary entry file.

The four-execution-mode pillar (daemon/standalone/client/piped) is half worth its cost. The standalone↔client transparent switching via breadcrumb + health probe is the product's signature trick and works well for mDNS (announce auto-heartbeats in client mode, unregisters on Ctrl+C). But the tax shows everywhere else: every command file carries two hand-written output arms (typed local vs. hand-parsed JSON client), UDP's standalone arms are admitted dead weight (4 of 5 bail; `udp bind` binds then immediately closes with a comment conceding "this isn't very useful"), no output ever indicates which mode actually ran, and piped NDJSON mode covers only mDNS. The mode magic also has an auth trap: explicit `--endpoint` produces an empty DAT token (commands/mod.rs:51-55) while the HTTP adapter requires X-Koi-Token on all mutations, so remote mutation commands fail; certmesh's `require_daemon` even sends the local breadcrumb's token to a remote endpoint. Security polish lags too: passphrases are read with echoing stdin reads, and `--passphrase` is a plain CLI arg. Overall: a thoughtfully designed CLI UX sitting on alpha-grade consistency, with duplication (windows.rs wiring, ceremony_cli.rs color/box helpers copy-pasted from certmesh.rs, status.rs capability blocks) as the dominant debt.

## Strengths

- Discoverability design is excellent: bare `koi` shows live daemon status + full command catalog with quick-start examples; bare `koi <domain>` shows a filtered catalog with curated workflow examples; `koi <command>?` opens a long-form detail page (crates/koi/src/main.rs:44-61, surface.rs:29-86, 425-540)
- Terminal-profile-aware rendering with graceful degradation: NO_COLOR, TERM=dumb, non-interactive stdout all fall back to PlainWriter/ASCII glyph presentations (surface.rs:301-317, command-surface/src/render/profile.rs)
- Consistent global flags across all domains: --json, --timeout, --endpoint, --standalone, -v/-vv, with env-var equivalents for every capability toggle (cli.rs:19-124); capability-disabled errors helpfully name both the flag and the env var (cli.rs:549-555)
- format.rs is a clean pure-function presentation module — all functions return String, 36 unit tests covering truncation, optional-field omission, tab-column counts (crates/koi/src/format.rs)
- Platform service management is production-quality: Windows SCM upgrade flow polls for deferred service deletion before recreating (windows.rs:100-114), sets recovery policy and per-port firewall rules with legacy-rule cleanup; Linux uses systemd Type=notify with sd_notify; macOS launchd with root:wheel ownership; all three check installed-state before demanding elevation
- Shared helpers genuinely reduce duplication where used: run_streaming() select-skeleton, with_mode()/with_mode_sync(), build_register_payload(), print_json() with graceful serialization failure (commands/mod.rs:81-219)
- Client-mode announce is thoughtful: auto-starts a heartbeat thread sized to the lease, stops it and unregisters on Ctrl+C (commands/mdns.rs:152-185)
- 52 CLI parse tests in cli.rs and 13 helper tests in commands/mod.rs give the clap surface real regression coverage
- Destructive operations gated by type-token confirmation (DESTROY/RESET) with non-tty refusal in factory-reset (factory_reset.rs:29-34)
- Graceful shutdown discipline duplicated correctly in both daemon paths: cancel → drain 500ms → join tasks → per-domain shutdown → 20s hard timeout → breadcrumb cleanup (main.rs:821-863)
- The interactive certmesh create wizard delegates all branching to a ceremony host, keeping the CLI a 'dumb render loop' — a clean protocol-driven UI pattern (commands/certmesh.rs:161-196, ceremony_cli.rs)

## Concerns

- surface.rs manifest has drifted from the real CLI with zero tests catching it: manifest name 'certmesh rotate-totp' (surface.rs:1245) vs actual command 'rotate-auth' (cli.rs:469) breaks `koi certmesh rotate-auth?`; descriptions/examples cite nonexistent flags — `--totp` (surface.rs:1011), `--exec` (surface.rs:1131), `--cidr` vs real `--subnet` (surface.rs:1233 vs cli.rs:461), `--process` health check (surface.rs:1609), `--include-logs` (surface.rs:1894,1914 — FactoryReset takes no args, cli.rs:151-152); dns lookup claims CNAME/TXT/SRV support but parser accepts only A/AAAA/ANY (surface.rs:1449 vs commands/dns.rs:41-49); factory-reset manifest claims 'Log files are preserved by default' but factory_reset.rs:50-58 remove_dir_all's the entire data dir including logs
- windows.rs run_service (windows.rs:379-847) duplicates ~400 lines of main.rs daemon_mode() wiring and has functionally drifted: Windows service mode never spawns the runtime orchestrator (main.rs:642) nor certmesh background tasks — renewal, roster sync, health heartbeat, failover (main.rs:801-809) — so the installed service silently lacks container orchestration and cert auto-renewal
- main.rs is 1,545 lines and contains real business logic despite the project rule 'main.rs contains zero business logic': a complete certmesh failover state machine with promote/demote decisions and CA mDNS announcement lifecycle (main.rs:917-1292)
- DAT auth trap in client mode: explicit --endpoint yields token: String::new() (commands/mod.rs:51-55) while all HTTP mutations require X-Koi-Token (adapters/http.rs:29,211-216), so remote mutations fail auth; certmesh require_daemon sends the LOCAL breadcrumb token to an explicit (possibly remote) endpoint (commands/certmesh.rs:89-94) — wrong credential to the wrong host
- Confirmation machinery in command-surface is dead code: Confirmation::gate()/requires_confirmation()/prompt_stdio() are never called outside the crate; certmesh destroy (certmesh.rs:1047-1074) and factory-reset (factory_reset.rs:20-44) hand-roll their own prompts, and --json silently bypasses the confirmation gate entirely for both destructive commands
- ceremony_cli.rs:22-120 copy-pastes the color module and visible_width/pad_visible/print_box box-drawing helpers from certmesh.rs:35-84,331-408 — the comment says '(reused from certmesh module)' but it is duplication, violating the project's own never-duplicate rule
- The 'format.rs is the single presentation layer' rule is fiction: 390 println!/eprintln! occurrences across 16 files; certmesh.rs alone has 103; every dual-mode command formats human output twice (typed local arm vs hand-parsed serde_json::Value client arm, e.g. commands/dns.rs:120-187)
- Passphrase handling: read via echoing stdin read_line (certmesh.rs:605-607, 892-895, 1095-1104 — no rpassword/termios), and --passphrase accepted as a plain CLI argument (cli.rs:419-421) leaking into shell history and process lists
- Standalone-mode semantics are inconsistent per domain: dns add/remove write local state files, udp bind binds then immediately cancels (commands/udp.rs:10-31, with an in-code admission it 'isn't very useful'), 4 of 5 udp commands bail, dns stop bails — and no command output indicates which mode executed, so users cannot tell whether a daemon or local fallback served them
- Stale documentation/agentic context: CLAUDE.md still documents crates/koi/src/openapi.rs which was deleted in commit b1a7262; .agentic/reference/utilities.md lists format.rs functions (certmesh_create_success, certmesh_status) that no longer exist; cli.md/manifest claim koi-runtime supports systemd/incus/kubernetes backends that only exist as enum variants
- status.rs offline_capabilities is ~107 lines of copy-pasted per-capability if/else blocks (commands/status.rs:80-187) that a 7-row table + loop would replace
- Hand-rolled utility code where crates exist: base32 decoder (certmesh.rs:484-503), duration parser (certmesh.rs:1130-1163), ANSI visible-width calculation duplicated twice — each a small correctness liability (visible_width mishandles non-SGR escapes and wide chars)
- JSON output is inconsistent across commands: some emit PipelineResponse envelopes, some pretty-print raw daemon JSON, some build ad-hoc serde_json::json! summaries (e.g. certmesh create --json returns {created,profile,ca_fingerprint} while non-json mode shows far more), making scripting unpredictable
- Standalone heavy-weight construction: `koi dns status` and `koi health add` spin up a full MdnsCore (worker thread, network sockets) just to read counts or write a config entry, then tear it down (commands/dns.rs:23-39, health.rs:19-46)
- The `koi <command>?` syntax is shell-fragile: unquoted `?` glob-expands in zsh (NOMATCH error) and in bash when a matching file exists — cute but will generate support noise

## Maturity signals

- Beta-grade for the mDNS/core path: 138 unit tests in the binary crate (52 parse tests in cli.rs, 36 format tests, dispatch/adapters covered), polished install/uninstall on three platforms, robust shutdown sequencing
- Alpha-grade for the broader CLI surface: surface.rs (1,931 lines, 54 commands) has zero tests and at least seven user-visible drift bugs; no integration tests exercise mode detection, client auth, or any end-to-end CLI flow
- Feasibility-test fingerprints: date-stamped version 0.2.202603241449; udp standalone arms stubbed with bail!s and an apologetic comment; KoiScope::Internal marked #[allow(dead_code)]; Confirmation framework built but never wired in
- Documentation outpaced by velocity: CLAUDE.md references deleted openapi.rs (removed in b1a7262), .agentic/reference/utilities.md lists functions that no longer exist in format.rs, the 'single presentation layer' and 'main.rs zero business logic' rules are both violated by current code — the agentic context layer is aspirational rather than enforced
- The Windows service / daemon_mode duplication with behavioral divergence (missing orchestrator + certmesh loops in service mode) is the kind of drift that indicates no cross-platform service-mode testing has occurred
- Recent git history shows active feature churn with same-day reverts (4426d77 reverting 4add29a), consistent with exploratory, single-maintainer development

## Shed or simplify

- Unify the daemon bootstrap: extract main.rs daemon_mode() core-and-adapter wiring into one shared function consumed by both main.rs and windows.rs run_service. This deletes ~400 duplicated lines and fixes the real bug that Windows service mode lacks the orchestrator and certmesh renewal/failover loops (windows.rs:379-847 vs main.rs:465-866)
- Move the certmesh failover state machine, renewal loop, roster sync, and health heartbeat (main.rs:917-1292, ~375 lines) into koi-certmesh behind a `spawn_background_tasks(core, mdns_events, cancel)` API. main.rs returns to pure wiring and the logic becomes testable in its domain crate
- Either generate surface.rs from clap or add a manifest-conformance test (every CommandDef name must round-trip through Cli::try_parse_from; every example command must parse). Without one of these, the 1,390-line hand-written build_manifest() will keep accumulating drift bugs like rotate-totp/rotate-auth. Alternative: collapse long_descriptions into clap `long_about` and let the catalog render from clap metadata, deleting most of surface.rs
- Delete or wire the Confirmation machinery: either call def.gate() in main.rs dispatch and remove the two hand-rolled prompts (certmesh.rs:1047-1074, factory_reset.rs:20-44), or remove Confirmation from command-surface entirely. Currently it is 60 lines of dead framework plus two divergent reimplementations
- De-genericize command-surface: it is a 1,038-line trait-parameterized framework (Category/Tag/Scope generics, two writer impls) with exactly one consumer. Folding a concrete renderer into the binary crate (or keeping the crate but dropping the generics) would shed several hundred lines and the Glyph/Scope indirection; KoiScope::Internal is already dead
- Extract one shared terminal-UI module (color enable/wrap, visible_width, pad_visible, print_box) used by certmesh.rs and ceremony_cli.rs — deletes ~100 duplicated lines and gives one place to fix the ANSI-width edge cases; or adopt a maintained crate (anstyle/console + unicode-width)
- Cut the dead standalone arms: udp bind/unbind/send/status/heartbeat and dns stop should use a single shared `requires_daemon(domain, cmd)` error helper instead of five with_mode boilerplate closures per file; standalone `udp bind` should be removed outright since it closes the socket immediately
- Table-drive offline_capabilities (status.rs:80-187): a const array of (name, disabled_flag) + one special-case for certmesh replaces ~100 lines with ~20
- Reduce dual-arm output duplication: have client arms deserialize daemon JSON into the same typed structs the local arms use (health.rs already does this for snapshots), then funnel both arms through one format function — per the project's own format.rs doctrine
- Fix the --endpoint token gap rather than carrying it: add KOI_TOKEN env / --token flag, and stop sending the local breadcrumb token to explicit endpoints (certmesh.rs:89-94). If remote administration is out of scope, reject --endpoint for mutating commands with a clear message
- Replace hand-rolled base32 (certmesh.rs:484-503) and duration parsing (certmesh.rs:1130-1163) with `data-encoding`/`base32` and `humantime` crates; adopt `rpassword` for all passphrase prompts and deprecate the plain --passphrase arg (keep an env-var escape hatch for scripting)
- Reconsider piped NDJSON mode: it covers only mDNS, has its own adapter + session-grace constant, and overlaps with `--json` streaming output. If usage is unproven, removing adapters/cli.rs (31 lines) plus its dispatch coupling simplifies the four-mode story to three
