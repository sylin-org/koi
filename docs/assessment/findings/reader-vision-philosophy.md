# Reader: vision-philosophy

## Summary

Koi presents itself as "the missing LAN toolbox" (README.md:11): a single cross-platform Rust binary that makes LAN services discoverable (mDNS), named (local DNS), trusted (private certificate mesh), watched (health), and reachable (TLS proxy), exposed to any language via HTTP/IPC/CLI. The stated audience is homelabs (5-15 machines), small businesses, and institutions "where there is a duty of care but not an IT department" (docs/archive/koi-spec.md:32-37). Two explicit philosophy statements anchor the project: "the secure path is the easy path" (koi-spec.md:38) and "the boundary is the local network. Everything that lives inside that boundary belongs in Koi" (koi-spec.md:1136). A third, implicit philosophy is the real feasibility test: building a multi-domain infrastructure product through agentic AI development — the repo carries a tool-agnostic AI context (.agentic/CONTEXT.md), an archived implementation guide literally signed "-Claude" (docs/archive/IMPLEMENTATION.md:780), three-voice documentation (wise mentor / precise technician / honest historian, CONTRIBUTING.md:127-135), 13 ADRs, and 9 defensive patent publications. The usage philosophy under test is roughly: capability monikers in one binary, runtime tunables instead of feature flags, bounded-context domain crates with commands/state/events faces, and "infrastructure that takes care of itself."

Scope evolved in three distinct waves over only ~7 weeks of intense development (first commit 2026-02-07, runtime adapter 2026-03-26, then near-silence until a single reverted commit 2026-05-10). Wave 1 (v0.1, Feb 7): an mDNS-only daemon wrapping mdns-sd with three thin adapters. Wave 2 (the archived koi-spec.md, self-labeled "Version 2.0.0-draft", Feb 2026): five capabilities (mdns, certmesh, dns, health, proxy) with a disciplined "What Not to Build" list rejecting container port forwarding, general monitoring, DNSSEC, and general reverse proxying. Wave 3 (Feb 16 - Mar 26): unplanned additions — koi-udp and embedded HTTP self-hosting (KOI-0001, explicitly motivated by a sibling project: "the ORCH offering orchestration suite in zen-garden requires containerized orchestrators to interact with the Garden mesh (UDP port 7184)"), dashboard + mDNS browser (KOI-0002), command-surface glyph crate, ceremony engine, envelope encryption/vault, and finally koi-runtime (ADR-013) which formally reverses ADR-010's rejection of in-binary Docker integration. Meanwhile, spec features were silently dropped: the Let's Encrypt Bridge Mode (koi-spec.md:162-170) was never implemented (the only "acme" in code is a test email string), and the spec's Cargo-feature-flag build plan (koi-spec.md:1104-1117) was reversed into a hard "never use #[cfg(feature)] for capabilities" rule.

The identity is coherent at the narrative level — the capability composition loop "discover → trust → name → watch → serve" (koi-spec.md:935-947) genuinely holds, and additions were each individually argued in ADRs/proposals. But the project now serves two diverging audiences: the human homelab operator (CLI, dashboard, ceremony wizards, compliance summaries) and the programmatic consumer (zen-garden via koi-embedded, containers via koi-udp/koi-runtime). The center of gravity has shifted toward the latter without the front-matter admitting it: GUIDE.md:3 still claims Koi handles "the two things every LAN needs" while listing nine capability guides; README's capability table omits runtime; and 32% of all Rust LOC now lives in certmesh alone. The nine defensive publications (all dated 2026-03-24) reveal where the author believes the novelty lies: six concern certmesh PKI (zero-config LAN certificate mesh, mDNS-based CA failover, heterogeneous envelope encryption with TOTP slots, ephemeral DH CA key transfer, scope-constrained enrollment, signed roster manifests), plus the ceremony engine, transport-adaptive leases, and UDP-over-HTTP/SSE bridging — explicitly NOT mDNS, which the README graciously credits entirely to mdns-sd ("Koi just gives it a friendly front door", README.md:168). In other words: the declared product is a LAN toolkit, but the claimed invention is a zero-config PKI with novel interaction machinery.

## Strengths

- Unusually explicit and quotable philosophy: 'the secure path is the easy path' (docs/archive/koi-spec.md:38) and a crisp scope rule 'the boundary is the local network' (koi-spec.md:1136), plus a 'What Not to Build' section with preserved rationale (koi-spec.md:1140-1158) — rare discipline for a hobby-scale project.
- Honest decision history: ADR-010 rejects Docker integration (2025-12-15), ADR-013 reverses it with explicit 'Supersedes: ADR-010' and a Negative-consequences section admitting the reversal (docs/adr/013-runtime-adapters.md:5,120-125). Archived specs carry clear 'superseded' headers (docs/archive/koi-spec.md:3-6).
- ADR-012 (consolidation roadmap) is a model of self-audit: executed blocks annotated DONE/SKIP/DEFERRED with commit hashes, including the integrity-preserving admission that ADR-011's cross-domain-import 'finding' was prospective, not real (docs/adr/012-consolidation-roadmap.md:124-127).
- The capability composition story is architecturally real, not marketing: 'container starts → mDNS announces → certmesh mints cert → DNS registers → health monitors → proxy terminates TLS' (koi-spec.md:937-947), and the integration points (SAN feedback loop, certmesh-SANs-as-DNS-records) are designed in, not bolted on.
- Defensive publications instead of patents is a coherent open-source stance, and they precisely document the novelty thesis with serious prior-art comparisons (Vault, EJBCA, AD CS, mkcert, step-ca) — e.g. docs/prior-art/defensive_pub_zero-config-lan-certificate-mesh.md sections 1-2.
- Tool-agnostic AI context (.agentic/CONTEXT.md bootstrapped from CLAUDE.md) with domain rules, reference tables, and 'Never Do / Always Do / Lessons Learned' sections — a genuinely transferable DX pattern for agentic development.
- Honest acknowledgment of dependency: README.md:166-168 credits mdns-sd with 'the heavy lifting', deflating any temptation to claim mDNS novelty.
- Three-voice documentation taxonomy (guides=mentor, reference=technician, ADRs=historian, CONTRIBUTING.md:127-135) and the docs largely deliver on it.

## Concerns

- Scope ratchet: each addition was individually argued (KOI-0001, KOI-0002, ADR-013) but the sum drifted from 'mDNS daemon' (Feb 7) to 15 crates, ~70 HTTP endpoints, ~50 CLI commands by Mar 26 — 7 weeks. The spec's own anti-scope rules were partially overridden: koi-udp/koi-runtime sit adjacent to the explicitly rejected 'container port forwarding trap' (koi-spec.md:1144-1146).
- Hidden second customer: koi-udp and embedded HTTP self-hosting exist for zen-garden's Garden mesh (docs/proposals/KOI-0001-embedded-http-and-udp-bridging.md:23 'Motivation: The ORCH offering orchestration suite... requires...'). This driver is invisible in README/GUIDE, so the public identity ('LAN toolbox for operators') and the actual roadmap driver (network substrate for a sibling orchestration project) diverge.
- Front-matter drift: GUIDE.md:3 says Koi handles 'the two things every LAN needs' while listing 9 guides; README.md capabilities table (lines 85-92) lists 6 capabilities but omits runtime, which only appears in the docs list (line 143); koi-spec.md calls itself 'Version 2.0.0' and shows 'koi v2.0.0' output while the project ships as v0.2 — version identity confusion.
- Documentation chronology is implausible: ADR-001 is dated 2025-01-15, ADR-007 2025-11-01, ADR-008 2025-12-01, ADR-010 2025-12-15 — all predating the repository's initial commit (2026-02-07, git log). Either retro-dated reconstruction or AI-fabricated dates; either way it undermines the 'honest historian' voice.
- ADR-011 asserts as fact ('In practice, koi-health imports four domain crates', docs/adr/011-security-architecture-overhaul.md:25) what ADR-012 later verifies never existed (012:124-127), and current Cargo.tomls confirm zero cross-domain deps. A security 'review of all 14 crates' that reports phantom findings is a credibility smell for the other 86 findings.
- Stated coding rules contradict reality: CONTRIBUTING.md:42-43 mandates 'No file over 300 lines / no function over 30 lines', yet koi-certmesh/src/lib.rs is 2,435 lines, koi/src/surface.rs 1,931, koi/src/main.rs 1,545 (and v0.1 guidance said main.rs over 60 lines means 'something is leaking up', IMPLEMENTATION.md:442).
- The spec's market 'catalyst' (CA/Browser Forum SC-081v3 47-day cert lifetimes, koi-spec.md:140-148) justifies certmesh only — it does not justify UDP bridging, runtime adapters, dashboards, or glyph systems; the original problem statement no longer covers the shipped surface.
- Momentum cliff: 176 commits total, dense Feb 7-Mar 26, then exactly one feature commit in the following ~6 weeks (2026-05-10) which was immediately reverted (4426d77). Consistent with 'feasibility test concluded' but the repo doesn't say so; an outsider cannot tell if this is paused, done, or abandoned.
- Spec promises never reconciled: Let's Encrypt Bridge Mode (koi-spec.md:162-170) is absent from code (grep: only a test email matches 'acme'); TPM sealing appears in the spec's security tables but the archived spec was never errata'd against what shipped — the only honest reconciliation lives scattered in ADR-012.

## Maturity signals

- Alpha-to-early-beta overall. Code-complete per ADR-012 ('All eight implementation phases (0-8) are code-complete... 807+ passing unit tests', docs/adr/012-consolidation-roadmap.md:9); today ~821 test functions (702 #[test] + 119 #[tokio::test]).
- Beta signals: executed consolidation roadmap with security hardening (panic elimination, Zeroizing wrappers, DAT auth, mTLS dual-port, X25519 promotion — ADR-011/012, commits 97f64e8, d223a35); published on crates.io as koi-net with CI enforcing clippy -D warnings; cross-platform service install (SCM/systemd/launchd); OpenAPI + Scalar docs; 9 polished user guides.
- Alpha/prototype signals: 'Blocks 5-6 introduce serde breaking changes with no migration path (acceptable for pre-1.0 project)' (ADR-012:322); FIDO2 unlock 'remains gated/broken until platform authenticator integration is solved' (ADR-012:324); koi-embedded had zero tests until the Mar 25 consolidation (ADR-012:39-42); ADR-011 found 17 CRITICAL security issues including RCE-via-set-hook on an unauthenticated 0.0.0.0 daemon as recently as 2026-03-18.
- Feasibility-test lifecycle pattern: entire arc from v0.1 daemon to 15-crate, 57.7k-LOC toolkit compressed into Feb 7 - Mar 26, 2026, followed by one reverted commit in 6 weeks — the cadence of an experiment that reached its demonstration point, not a maintained product.
- The 9 defensive publications (all dated 2026-03-24, one day before the consolidation ADR) signal the author treating the design as finished intellectual output worth fencing — a milestone marker more than a growth signal.
- Project self-describes its modes-of-use maturity unevenly: mDNS path is battle-shaped (leases, draining, reaper, admin surface); certmesh is feature-deep but flagged with deferred FIDO2 and breaking-change tolerance; runtime adapter landed in a single day (2026-03-26, commits bbab222 through e1938a6) and is the least proven.

## Shed or simplify

- command-surface crate (1,038 LOC) + koi/src/surface.rs (1,931 LOC manifest): a glyph/category/scope metadata system with NerdFont→emoji→ASCII degradation tiers (docs/adr/007-command-surface.md) for a CLI whose help is already owned by clap. ADR-012 Appendix B even keeps dead code (KoiScope) for 'future command-surface scope filtering'. Candidate: fold the manifest into the binary crate, drop glyph tiers, delete the standalone crate.
- koi-udp (686 LOC + guide + defensive pub): exists solely because zen-garden's Garden mesh needed UDP from bridge-networked containers (KOI-0001 Motivation). It is the most foreign limb relative to 'the missing LAN toolbox'. Candidate: extract to a zen-garden-side satellite or explicitly re-label it as the container-bridging story's edge case; at minimum stop presenting it as a peer pillar of mDNS/DNS/certmesh.
- Dual orchestrators: koi-embedded (5,801 LOC) re-implements capability wiring that crates/koi/src/main.rs (1,545 LOC) also performs. Two builders for the same composition is structural debt. Candidate: make the binary a thin shell over the embedded facade (one wiring path), which would also shrink main.rs toward its own stated ideals.
- Certmesh institutional surface: trust profiles, compliance summaries, operator attribution, enrollment windows, audit-log voice aimed at 'schools, libraries, clinics' (koi-spec.md:36) — an audience there is no evidence of. Certmesh is 18,610 LOC, 32% of the codebase, with 19 CLI subcommands and 19 HTTP endpoints. Candidate: keep the JustMe profile path lean and demote Team/Organization machinery (compliance, attribution, windows) to a clearly-staged tier, or accept that Koi is really 'certmesh + supporting cast' and rewrite the front-matter accordingly.
- Reconcile or delete dead spec promises: Let's Encrypt Bridge Mode (koi-spec.md:162-170) was never built — add an erratum to the archived spec or an ADR declaring it dropped, so the archive stops advertising vaporware.
- Fix the small identity drifts cheaply: GUIDE.md:3 ('the two things every LAN needs'), README capabilities table missing runtime (README.md:85-92), 'v2.0.0' strings in the archived spec vs v0.2 reality, and the pre-repo ADR dates (001/007/008/010) — either annotate as reconstructed or correct.
- CONTRIBUTING.md size rules (300-line files, 30-line functions, CONTRIBUTING.md:42-43) are violated by an order of magnitude in 13+ files; either enforce via CI on new code or rewrite the rule to match the project's real shape — rules that are visibly ignored teach contributors to ignore the rest.
- Dashboard/browser ambitions (KOI-0002 phases 7.2-7.4: browser actions, more pages, WebSocket upgrade — deferred in ADR-012:298-307): explicitly freeze rather than leave open-ended; the embedded dashboard is presentation, not a pillar.
- ADR-011's unverified findings: re-audit the remaining CRITICAL/HIGH claims with the same rigor ADR-012 applied to Root Cause 4, and annotate ADR-011 with which findings were real — otherwise the project's flagship security document contains known fabrications.
