# Reader: docs-onboarding

## Summary

Koi's documentation corpus is large (~24k lines of markdown vs ~58k lines of Rust) and unusually well-written for a feasibility project, but it has bifurcated into two eras: pre- and post-ADR-011 (the 2026-03-18 security overhaul). The overhaul moved the HTTP adapter to loopback-only binding (crates/koi/src/adapters/http.rs:236-239) and added a mandatory Daemon Access Token on all mutation endpoints (http.rs:453-488), yet the primary onboarding surface — README.md, GUIDE.md, and the 914-line CONTAINERS.md — still documents the old world: CONTAINERS.md:31 claims "Koi binds to 0.0.0.0:5641... Containers can reach it without any extra configuration," and README's headline `curl -X POST .../v1/mdns/announce` example now returns 401. The project's differentiating pitch (containers gain LAN capabilities via plain HTTP) is therefore broken as documented on native Linux Docker, and partially broken everywhere. Only docs/guides/certmesh.md reflects the new dual-port/token architecture. The new-user CLI path (download release binary → `koi mdns discover`) does work: release engineering is genuinely mature (6 cross-compiled targets, checksums, crates.io publishing), though README claims Rust 1.75+ while Cargo.toml pins rust-version 1.92.

Accuracy is bimodal per document, not uniformly poor. Spot-checks: mdns.md and udp.md are excellent and match code (error tables match koi-common/src/error.rs exhaustively; UDP shapes match koi-udp/src/lib.rs), and docs/reference/cli.md tracks cli.rs closely including the new factory-reset. But docs/reference/http-api.md (the "authoritative spec") has a fabricated UDP section (wrong bind body, wrong response field names, claims 30s expiry vs the code's 300s default), never mentions the x-koi-token auth it now requires, and misstates CORS. Three docs plus the code give four different stories on certmesh trust-profile defaults (certmesh.md vs ceremony-protocol.md:117-124 vs koi-certmesh/src/profiles.rs:34-46), and envelope-encryption.md claims an Ed25519 CA with 90-day member certs while code uses ECDSA P-256 with 30-day certs (koi-crypto/src/keys.rs:134, koi-certmesh/src/ca.rs:16). system.md declares factory-reset "planned but not yet implemented" while it ships fully implemented (commands/factory_reset.rs, 114 lines). embedded.md contains Rust samples that cannot compile (UdpSendRequest with a nonexistent binding_id field).

The .agentic/ AI-context system is a thoughtful, novel DX artifact — tool-agnostic bootstrap, explicit three-voice documentation philosophy, multi-assistant config table — but it is structurally a third copy of the reference docs and is drifting independently: utilities.md's error-code table is missing 10 of 30 codes present in error.rs, CONTEXT.md omits factory-reset and lists bollard 0.18 after the 0.20 upgrade, and .agentic/README.md:56 mandates CONTEXT.md stay under 100 lines while it is 376. CONTRIBUTING.md's own rules ("No file over 300 lines," "avoid async-trait") are contradicted by surface.rs (1,931 lines) and koi-runtime's async-trait dependency. architecture.md's crate-size table understates reality 4x (claims koi bin ~3,500 lines, actual 14,077; koi-certmesh ~4,500, actual 18,610) and omits four binary-crate modules added since (orchestrator.rs, integrations.rs, mtls.rs, factory_reset.rs). The meta-documentation (prior-art/ 6,905 lines of defensive publications, proposals/ 4,262, archive/ 3,731) outweighs the user-facing guides+reference (3,459 lines) by 4:1, which is the clearest sign of documentation over-investment relative to an alpha codebase whose behavior changed underneath the docs within the last two weeks of commits.

## Strengths

- Guide prose quality is exceptional and consistent: every guide opens with a problem statement, a 'When to use' paragraph, and conceptual depth (lease lifecycle in docs/guides/mdns.md:317-338, three record sources in docs/guides/dns.md:11-21, ceremony bag model in docs/reference/ceremony-protocol.md:19). The 'three voices' philosophy (mentor/technician/historian) is explicitly codified in CONTRIBUTING.md:126-136 and .agentic/README.md:74-79 and actually followed.
- Release/install path is real, not vaporware: .github/workflows/release.yml builds 6 targets (x86_64/aarch64 for Linux-musl, macOS, Windows) with sha256 checksums and publishes 12 crates to crates.io with hand-rolled rate-limit handling; 12 git tags confirm releases shipped. `cargo install koi-net` is accurate (crates/koi/Cargo.toml package name = koi-net).
- Spot-checked accuracy of the best docs is high: docs/guides/mdns.md error table matches koi-common/src/error.rs exactly (including ambiguous_id, already_draining, resolve_timeout); CLI flags --timeout/--ip/--record-type all match crates/koi/src/cli.rs; docs/guides/udp.md request/response shapes match koi-udp/src/lib.rs:44-84 including the 300s default lease; docs/reference/cli.md correctly documents factory-reset and runtime backend selection.
- ADRs are substantive and unusually honest: ADR-011 catalogs 87 security findings against the project's own code with root-cause analysis (RCE via set-hook, broken FIDO2 slot derivation); ADR-010 documents a rejected proposal; ADR-012 is a dated, executed consolidation roadmap. This is historian-grade decision documentation rare at this maturity level.
- docs/ hierarchy is disciplined: guides/reference/adr/qa/proposals (with implemented/ subdir)/archive separation; retired docs actually moved to archive/ rather than rotting in place.
- The .agentic/ concept itself — tool-agnostic AI context with CLAUDE.md/.cursorrules/etc. as thin bootstrappers pointing at one source — is ahead of common practice and includes a maintenance philosophy ('if AI already does something correctly, remove the rule', .agentic/README.md:58).
- Onboarding first-success via CLI is genuinely fast: `koi mdns discover` with zero config is a strong 30-second win, and GUIDE.md leads with it correctly.
- docs/guides/runtime.md (rewritten 2026-03-26) is current and matches the new orchestrator architecture, including honest 'Planned' status for systemd/Incus/Kubernetes backends that exist only as enum stubs (koi-runtime/src/lib.rs:252-258).

## Concerns

- BROKEN HEADLINE USE CASE: CONTAINERS.md:31 claims Koi binds 0.0.0.0:5641 and 'containers can reach it without any extra configuration'; code binds 127.0.0.1 only with an explicit comment (crates/koi/src/adapters/http.rs:236-239). All Linux container paths documented in CONTAINERS.md:110-118 (172.17.0.1 bridge gateway, host-gateway) cannot reach a loopback-bound daemon. The project's core container pitch in README.md:71-79 fails as documented.
- UNDOCUMENTED AUTH BREAKS ALL WRITE EXAMPLES: DAT middleware rejects every non-GET request lacking x-koi-token (crates/koi/src/adapters/http.rs:453-488). README.md:34, CONTAINERS.md (dozens of POST examples), docs/guides/udp.md, and docs/reference/http-api.md show unauthenticated curl POSTs that now return 401. The token is mentioned in exactly one user doc (certmesh.md network table); http-api.md — self-described authoritative reference — never mentions authentication, and claims permissive CORS while code restricts origins to localhost (http.rs:230-233, contradicting mdns.md:311 'CORS is enabled for all origins').
- REFERENCE DOCS CONTRADICT EACH OTHER AND CODE: envelope-encryption.md:83-85 says Ed25519 CA, 90-day member certs; code is ECDSA P-256 (koi-crypto/src/keys.rs:134-140) with 30-day certs (koi-certmesh/src/ca.rs:16); certmesh.md:277-279 agrees with code. ceremony-protocol.md:117-124 profile table (just_me=Closed enrollment, my_team=no approval, passphrase unlock) contradicts both certmesh.md:56-61 and code (profiles.rs:34-46: JustMe/MyTeam open + MyTeam requires approval + MyTeam auto-unlocks). Four sources, four stories.
- http-api.md UDP section appears written from imagination: bind body `{"addr": "0.0.0.0:9999"}` vs actual UdpBindRequest{port,addr,lease_secs} (koi-udp/src/lib.rs:53-63); response `{"sent": N}` vs guide's correct `bytes_sent`; 'expire after 30 seconds' vs 300s default. Meanwhile udp.md:84 uses POST for heartbeat where the route is PUT (koi-udp/src/http.rs:92).
- system.md:98-100 says 'koi factory-reset is planned but not yet implemented' while it is fully shipped (cli.rs:151, commands/factory_reset.rs 114 lines) — and the same doc's closing line (system.md:187) assumes it exists. dns.md:163 references nonexistent `koi mdns status` (actual: `koi mdns admin status`).
- embedded.md code samples cannot compile: UdpBindRequest struct literal missing required fields (embedded.md:101-103) and UdpSendRequest given a nonexistent binding_id field (embedded.md:104-108 vs koi-udp/src/lib.rs:44-49). No doctests or compile-checked snippets anywhere.
- architecture.md metrics stale by 2-4x: claims crates/koi ~3,500 lines (actual 14,077), koi-certmesh ~4,500 (actual 18,610), 'each adapter roughly 150 lines' (http.rs is 834); binary layout (architecture.md:101-131) omits orchestrator.rs (458 lines), integrations.rs (284), adapters/mtls.rs (189), factory_reset.rs. Both it and .agentic/CONTEXT.md still place dashboard/browser in the binary crate while code serves them from koi_common::dashboard/browser with HTML in crates/koi-common/assets/ — koi-common's 'shared kernel' label no longer matches its contents (563-line browser.rs, firewall.rs, net.rs).
- .agentic/ drift: utilities.md error-code table missing 10 of 30 ErrorCode variants in error.rs (ambiguous_id, parse_error, revoked, approval_*, etc.); CONTEXT.md command list omits factory-reset; lists bollard 0.18 after commit c23af0a upgraded to 0.20; .agentic/README.md:56 mandates CONTEXT.md <100 lines, actual 376.
- CONTRIBUTING.md rules contradicted by the codebase it governs: 'No file over 300 lines' (surface.rs 1,931; main.rs 1,545; cli.rs 1,431; ceremony.rs 1,296), 'avoid async-trait' (koi-runtime depends on it), README says Rust 1.75+ while Cargo.toml workspace pins rust-version = 1.92 — build-from-source per README instructions fails on 1.75.
- CI doc/workflow mismatch: .github/workflows/qa.yml runs `bash tests/integration.sh --no-build` on Linux/macOS, but tests/ contains only integration.ps1 and concurrency.ps1 — the Linux/macOS QA jobs reference a nonexistent script.
- ADR-011 status still 'Proposed' though its dual-port/DAT/mTLS design is implemented and shipped — ADR status hygiene undermines the 'honest historian' contract.
- No user-facing security model document exists despite the security overhaul being the largest recent change: bind addresses, token lifecycle, breadcrumb permissions, mTLS port, and threat model are only discoverable via ADR-011 and one table in certmesh.md.

## Maturity signals

- Alpha, not beta: the primary documented integration path (HTTP from containers) does not work against the current binary — docs describe a product that existed before 2026-03-18 (ADR-011 implementation). A real user following README/CONTAINERS today hits 401s and connection-refused.
- Release engineering is beta-to-production grade (cross-platform binaries, checksums, crates.io publish automation with rate-limit recovery, CI on 3 OSes with clippy -D warnings and fmt gates) — well ahead of doc accuracy and feature stability.
- Versioning is pre-stability: timestamp versions (0.2.202603241449), releases cut on every push to main with tag deletion/re-tagging (release.yml:189-196), no changelog discipline beyond a template.
- Documentation volume signals aspiration beyond feasibility-test scope: 9 defensive-publication prior-art documents (6,905 lines), 13 ADRs, compliance endpoints, FIDO2 research — patent-adjacent and enterprise-flavored artifacts atop an unaudited homelab tool.
- ADR-012 self-reports '807+ passing unit tests' and 8 completed implementation phases, and the consolidation roadmap was executed within days (status: 'executed 2026-03-25') — high velocity, single-author cadence with AI assistance; docs lag code by roughly 1-3 weeks consistently (most guides dated Feb 16, security overhaul Mar 18, only runtime.md and certmesh.md refreshed since).
- Three of six advertised runtime backends (systemd, Incus, Kubernetes) are enum stubs returning BackendUnavailable (koi-runtime/src/lib.rs:252-258); cli.rs:86 help text advertises all six while docs honestly list them as planned — code surface outruns implementation.
- The existence of docs/qa/validation-suite.md (573 lines) and weekly scheduled integration CI shows QA intent, but the broken integration.sh reference means the cross-platform QA matrix has never run green as configured.

## Shed or simplify

- Collapse .agentic/reference/ into docs/reference/: api-endpoints.md (303 lines) is a hand-maintained subset of http-api.md+cli.md, and utilities.md (486 lines) duplicates architecture.md's type inventories — both have drifted independently from code AND from each other. Keep .agentic/ for rules only (CONTEXT.md + rules/) and link to docs/reference as the single lookup source; this halves the drift surface.
- Generate instead of hand-write the duplicated API surface: the project already builds a full OpenAPI spec from utoipa (GET /openapi.json) and clap definitions — http-api.md endpoint tables and cli.md flag tables could be generated artifacts or CI-diffed against the spec, eliminating the class of error found in the UDP section.
- Move docs/prior-art/ (6,905 lines) and docs/archive/ (3,731 lines) out of the main repo (separate repo, wiki, or release-notes archive). Together they are 50% of docs/ by volume and zero percent of user value; they inflate the maintenance perception and clone size.
- Write ONE post-ADR-011 'Networking & security model' page (bind addresses, x-koi-token lifecycle, breadcrumb file, CORS policy, mTLS port, threat model) and make README, GUIDE.md, CONTAINERS.md, http-api.md, and mdns.md reference it — currently the security model is documented only in a Proposed-status ADR and one certmesh.md table, while five user docs contradict it.
- Rewrite or quarantine CONTAINERS.md until the container path actually works: either add a `--http-bind` flag (with token guidance for containers) or mark the guide as targeting a previous release. Shipping 914 lines of detailed Compose/K8s patterns that fail on first curl is worse than a short honest page.
- Single source for certmesh profile defaults: the profile table exists in certmesh.md, ceremony-protocol.md, and implicitly in profiles.rs with three different contents. Keep it in one reference table; better, derive the doc table from a unit test that asserts profiles.rs behavior.
- Drop or fix the stale CONTRIBUTING.md size rules (300-line files, 30-line functions) — they are violated by an order of magnitude (surface.rs 1,931 lines) and erode trust in the rest of the document; replace with the limits actually enforced.
- Trim the .agentic/CONTEXT.md command/type inventory (376 lines, target <100 per its own README): the full CLI and type listings duplicate cli.md/utilities.md and are the fastest-rotting sections; keep boundary rules and never-do lists, link out for inventories.
- Fix or remove the qa.yml Linux/macOS jobs referencing nonexistent tests/integration.sh — a scheduled workflow that can never pass is pure noise.
- Mark ADR-011 as Accepted/Implemented and add a one-line status to ADR-012's blocks — the ADR set is the project's best asset and cheap status hygiene keeps it trustworthy.
