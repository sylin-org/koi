# Reader: quality-infra

## Summary

Koi's engineering infrastructure is far more elaborate than a typical hobby project — 3 GitHub workflows (CI, QA, Release), a 3,496-line PowerShell integration harness, ~800 unit tests, multi-platform release builds for 6 targets with checksums, and a crates.io publishing pipeline with hand-rolled rate-limit handling — but significant parts of that machinery are silently or visibly broken, and nobody is watching. The weekly scheduled QA workflow has failed every single run since at least 2026-03-30 (10+ consecutive failures) because qa.yml still invokes tests/integration.sh and tests/concurrency.sh, which were deleted on 2026-02-12 (commit fe8151c "remove .sh scripts"). The crates.io publish job reports green but is silently broken: the `cargo publish | tee` pipeline in release.yml:289 runs without pipefail, so failures are swallowed — the installable binary crate `koi-net` has been stuck on crates.io at 0.2.202602121703 (Feb 12) and koi-embedded at Feb 16, while koi-udp, koi-runtime, and command-surface have never been published at all (404 on crates.io) because they're missing from the CRATES list (which still publishes lib crates: koi-common etc. are current at 0.2.202603241449). `cargo install koi-net` — the documented install path in README.md:125 and the release template — would build a 4-month-stale binary against newer 0.2.x lib crates (loose "0.2" internal version requirements), likely failing to compile.

The CI workflow proper (ci.yml) is genuinely solid for the project's stage: build+test on ubuntu/windows/macos, clippy -D warnings, fmt --check, --locked builds, rust-cache. The release process is unusual: every push to main triggers a full 6-target release, with tags derived from a timestamp version (0.2.YYYYMMDDHHMM read from Cargo.toml, which is mutated by build.ps1 on the developer's machine). The release job deletes and re-creates existing tags/releases ("clean slate", release.yml:189-196), making release history mutable. Versioning carries zero compatibility semantics — minor is frozen at 2, patch is a wall-clock timestamp, there is no CHANGELOG, no SemVer discipline, and internal crate deps use loose "0.2" requirements that span breaking API changes (a security overhaul happened between published versions).

Test depth is heavily lopsided and tracks where the feasibility energy went: koi-certmesh has 264 unit tests over 18.6k LOC and koi (binary) 149 over 14k LOC, but the newer domain crates are nearly untested — koi-udp 3 tests, koi-proxy 5, koi-health 5, koi-dns 10, command-surface 0. The Docker runtime backend's only integration tests are all #[ignore] (require a live Docker daemon) and no CI job runs --ignored, so koi-runtime's actual backend is never exercised in CI. End-to-end coverage exists only on Windows (integration.ps1, ~170 assertion sites covering HTTP/SSE/pipe/certmesh/lease-expiry/Tier-3 service lifecycle); the Linux/macOS equivalents were deleted and never replaced, so daemon E2E, systemd/launchd service install, and the dashboard/mdns-browser UIs are untested on 2 of 3 platforms (dashboard untested everywhere). The validation suite (docs/qa/validation-suite.md, 573 lines) is a thoughtful manual-acceptance spec with phase gates — but every gate checkbox is still unchecked, including the "product promise" E2E-05 test. Activity stopped abruptly: 176 commits by a single author between Feb 7 and May 10, 2026, with nothing pushed since Mar 26 (the only local commits since are a feature and its own revert). The repo has 0 stars, 0 forks, 0 issues — the infrastructure is release-shaped, but there has been no actual release audience, and the maintenance loop (watching CI, publishing, versioning) has already decayed. This is consistent with the stated identity: a feasibility test of a philosophy, wearing production-release clothing it doesn't yet need.

## Strengths

- Real 3-OS CI matrix with clippy -D warnings, cargo fmt --check, and --locked builds (.github/workflows/ci.yml) — better hygiene than most solo OSS projects
- Substantial Windows E2E harness: tests/integration.ps1 (3,496 lines, ~170 Pass assertion sites) covers daemon startup, HTTP/SSE/pipe adapters, certmesh ceremonies, lease expiry timing (12s/40s waits), graceful shutdown, and Tier-3 Windows service lifecycle
- ~800 unit tests workspace-wide (validation-suite.md:8 claims '779+ passing'); koi-certmesh alone has 264, koi-crypto 99 — the security-critical crates are the best-tested
- 6-target release builds (linux-musl x2, darwin x2, windows x2) with SHA-256 checksums and a templated release page (.github/release-template.md); 12 releases shipped Feb 8 – Mar 26
- Cargo.lock committed and used (--locked) consistently across CI, release, and publish — reproducible builds for a binary-shipping project
- docs/qa/validation-suite.md is a genuinely good artifact: 573 lines of ID'd test cases with automated/manual classification, negative-tests section (NEG-01..12), performance baselines (binary <30MB, idle <50MB RSS, startup <2s), and phase gates
- CONTRIBUTING.md is opinionated and specific (no anyhow in domain crates, no tower/ORM/gRPC, file/function size limits, logging level contract) — encodes the project philosophy in reviewable rules
- Commit hygiene is decent: 134/176 commits (76%) follow conventional-commit format, with disciplined fix(security)/fix(safety) review-loop series; single author, clean working tree
- Per-crate crates.io metadata is mostly complete (descriptions everywhere, READMEs in 13/15 crates, keywords/categories on the binary crate, dual MIT/Apache-2.0 license)
- Workspace-level dependency pinning in root Cargo.toml [workspace.dependencies] — single version pin per external dep, no version drift between crates

## Concerns

- Scheduled QA workflow 100% red for 2.5+ months: every weekly run since 2026-03-30 failed because .github/workflows/qa.yml:21,43,65,76,87 invoke tests/integration.sh and tests/concurrency.sh, deleted in commit fe8151c (2026-02-12). 5 of 8 QA jobs cannot even start. Nobody noticed or fixed it — CI-as-theater risk
- Silently broken crates.io publishing: release.yml:289 `cargo publish -p $crate --locked 2>&1 | tee ... || {handler}` runs in GitHub Actions default shell WITHOUT pipefail, so tee masks every publish failure and the error handler (incl. `exit 1` at line 316) never fires. Evidence: publish job green on Mar 26 while koi-net remains at 0.2.202602121703 (Feb 12) and koi-embedded at 0.2.202602151054 (Feb 16) on crates.io
- Publish CRATES list (release.yml:218-231) omits koi-udp, koi-runtime, and command-surface entirely — yet koi-net and koi-embedded depend on all three (crates/koi/Cargo.toml:29-30,53; crates/koi-embedded/Cargo.toml). Those two crates can never publish again until the list is fixed; the list also still documents a stale layer model ('koi-client # Layer 5: depends on health + mdns')
- `cargo install koi-net` (recommended in README.md:125 and .github/release-template.md:10) is a trap: the Feb 12 koi-net source will resolve loose '0.2' internal deps to the Mar 26 published libs, which underwent a breaking security-architecture overhaul (ADR-011) in between — likely compile failure or behavioral mismatch
- Versioning scheme 0.2.YYYYMMDDHHMM (workspace Cargo.toml:22, generated by build.ps1:39-50 which permanently mutates Cargo.toml on the dev machine) conveys no compatibility information; minor frozen at 2 across breaking changes; no CHANGELOG anywhere in the repo; version.json holds only major/minor + a one-line description reused for every release
- Mutable release history: release.yml:189-196 deletes existing releases, deletes and force-re-pushes tags on every main push — same tag can point to different commits over time; releases also rebuild on docs-only commits (no path filtering), burning 6-platform CI on every push
- Test coverage is inverted relative to risk surface for newer domains: koi-udp 3 unit tests / 686 LOC, koi-proxy 5 / 954, koi-health 5 / 1,070, koi-dns 10 / 1,678, command-surface 0 / 1,038 — the TLS-terminating proxy and DNS resolver are among the least-tested crates
- koi-runtime's Docker backend integration tests are all #[ignore] (crates/koi-runtime/tests/docker_integration.rs:24,73,106,149,272,299) and no CI job passes --ignored or provisions Docker — the bollard-based backend is never executed in CI
- Linux/macOS have no E2E coverage at all since the .sh harnesses were deleted: daemon lifecycle, systemd/launchd install (koi install/uninstall), breadcrumb discovery, and UDS pipe adapter are validated only by unit tests on 2 of 3 supported platforms; docs/qa/test-plan.md:39,50,90 still references the deleted scripts
- CI/local test invocation mismatch: build.ps1:74-81 runs koi-certmesh tests with --test-threads=1 ('Certmesh tests failed' otherwise), but ci.yml:33 runs plain `cargo test --locked` — either the serialization workaround is stale or CI is exposed to the isolation races referenced in commits 058b087/11b8311/dc843b6
- No MSRV verification: rust-version=1.92 declared (Cargo.toml:25) after a bump/revert/re-bump dance (e53878c → 94b0518 → 2f1ab3a), but CI only tests stable — MSRV is an unchecked claim
- No supply-chain or security automation despite the project being security-themed (private CA, cert mesh): no cargo-audit/cargo-deny job, no Dependabot config, no SECURITY.md, no signed binaries (sha256 only, no Sigstore/minisign), no SLSA provenance
- CONTRIBUTING.md contradicts the codebase: 'No file over 300 lines' (line 41) vs koi-certmesh/src/lib.rs 2,435 lines, koi/src/surface.rs 1,931, koi/src/main.rs 1,545; 'Avoid async-trait' (line 55) vs koi-runtime/src/backend.rs:38 and docker.rs:132 using #[async_trait]
- All validation-suite phase-gate checkboxes (validation-suite.md Checklist Summary) remain unchecked, including E2E-05 ('This is the product promise. If this test fails, the product isn't done.') — manual acceptance was specified but never recorded as performed
- GitHub repo description still says 'Local mDNS service discovery for everyone' while the project is now an 11-domain network toolkit; 0 stars / 0 forks / 0 issues; nothing pushed since 2026-03-26 and the only post-Mar-26 local commits are a feature plus its immediate revert (4add29a/4426d77) — momentum has stalled with broken automation left in place

## Maturity signals

- Alpha, trending toward 'paused feasibility prototype with beta-shaped tooling': the infrastructure (3-OS CI, 6-target releases, crates.io pipeline, QA cron) imitates a mature project, but the feedback loops that define maturity — watching scheduled CI, verifying publishes landed, checking acceptance gates — are absent or decayed
- Prototype signal: 176 commits, single author, 13 weeks of activity (2026-02-07 → 2026-05-10), then silence; last push to origin 2026-03-26; final local commits are an experiment and its revert
- Beta signal: 779+ unit tests, serde round-trip discipline, dedicated security-review commit campaigns (≈15 fix(security)/fix(safety) commits), 13 ADRs — code-quality practices exceed typical alpha
- Prototype signal: release automation fires on every push to main with mutable tags and timestamp versions — a cadence designed for a solo developer's iteration loop, not for downstream consumers
- Alpha signal: distribution is GitHub Releases archives only — no winget/scoop/brew/apt/AUR packaging, no install script with checksum verification, no binary signing; the one package-manager path (cargo install koi-net) is broken/stale
- Prototype signal: scheduled QA red for ~10 consecutive weeks without remediation, and the crates.io publish silently failing since February — automation exists but is unmonitored, meaning there is effectively no release process being operated right now
- Alpha signal: test pyramid is unit-heavy with one platform of E2E (Windows); manual acceptance suite is well-specified but 0% executed (all gates unchecked)
- Zero external adoption signals: 0 stars, 0 forks, 0 issues, 30-933 crates.io downloads per crate (mostly mirror/bot traffic levels)

## Shed or simplify

- Delete or radically shrink the crates.io publishing ambition: 15 crates with a hand-rolled 125-line rate-limit-aware bash publisher (release.yml:203-326) is the single most over-built piece of infrastructure, and it's silently broken. For a feasibility project, publish either nothing or just koi-net (binary) — or use cargo-workspaces/release-plz instead of bespoke bash. If kept, the minimum fixes are: add `set -o pipefail` (or drop the `| tee`), add koi-udp/koi-runtime/command-surface to the list, and verify published versions post-run
- Fix or delete qa.yml: 5 of 8 jobs reference deleted .sh scripts and have failed weekly for months. Either port integration.ps1 to run cross-platform under pwsh on ubuntu/macos runners (pwsh is preinstalled on all GitHub runners — the .sh twins were never necessary), or cut qa.yml to the 3 jobs that can actually run (integration-windows, service-windows, concurrency-windows)
- Stop releasing on every push to main. Move release.yml to tag-push or workflow_dispatch trigger; this eliminates the tag-delete/re-create mutation hack (release.yml:189-196), stops rebuilding 6 targets for docs commits, and makes 'release' a deliberate act with a CHANGELOG entry instead of a side effect
- Replace timestamp versioning with plain pre-1.0 SemVer (0.3.0, 0.3.1...). The version.json + build.ps1 Cargo.toml-mutation machinery (build.ps1:36-50) exists solely to support the timestamp scheme; ordinary git tags + a one-line bump make build.ps1 mostly deletable (it otherwise duplicates `cargo build && cargo test`)
- Drop the 'manual acceptance checklist' pretense or automate the top slice: validation-suite.md's 200+ manual test cases will realistically never be executed by hand (0 gates checked in 3 months). Keep it as a design spec, but extract the ~20 highest-value automatable cases into the integration harness and delete the checkbox-gate framing
- tests/integration.ps1 at 3,496 lines in one file is its own maintenance hazard (the project's CONTRIBUTING.md caps source files at 300 lines). Split into per-domain script modules or convert the HTTP-level tests into Rust integration tests in the binary crate (where they'd run on all 3 OSes in the existing ci.yml matrix for free), keeping PowerShell only for OS-service Tier-3 tests
- Retire dist/ai-context (12 generated markdown bundles) and scripts/AiDocDist.ps1 (460 lines) from the repo's operational surface if they aren't actively consumed — generated doc-distribution tooling for a project with zero external consumers is pure carrying cost
- Reconcile CONTRIBUTING.md with reality or with intent: either enforce the 300-line/30-line/no-async-trait rules (CI lint) or delete the claims — rules that the codebase visibly violates (koi-certmesh/src/lib.rs 2,435 lines; koi-runtime uses async-trait) train contributors to ignore the whole document
- Add the three cheap credibility items instead of more bespoke automation: a CHANGELOG.md, a cargo-audit/cargo-deny CI job (one action each), and an MSRV check job (dtolnay/rust-toolchain@1.92) — these replace far more elaborate broken machinery with low-maintenance signals
